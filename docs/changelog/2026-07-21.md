**踩过的坑 —— 绝对不要再犯**：

> - **现象**：`generate-api.sh` 跑完后 `git status` 冒出来 300+ 个从没见过的 `??` 文件，遍布 blog/chat/iam/misc/monitoring/sdk/system/task/video 几乎所有域。**根因**：脚本里 `goctl api go` 没传 `-style` 参数，本机装的 goctl 1.10.1 默认用无下划线命名（`blogarticlecreatehandler.go`），而项目历史上所有已提交文件都是下划线命名（`blog_article_create_handler.go`，大概率是早期用的 goctl 版本默认风格不同）。goctl 按文件名判断"是否已存在、要不要生成"，两种命名一对不上，就把全项目当成"文件不存在"重新生成了一遍，跟已有文件同包同名重复声明，编译直接报错。**正确做法**：`generate-api.sh` 里 `goctl api go` 必须显式加 `-style go_zero`，不能依赖 goctl 的默认值（不同版本默认值可能不同）；已经在这次改动里修了。如果之后升级 goctl 或换机器又出现类似"多出一堆同名文件"的情况，第一反应是检查命名风格，不是慌着删文件。
> - **现象**：同一次 `generate-api.sh` 还多生成了 11 个 `internal/middleware/*_middleware.go`（下划线命名）和一个 `internal/svc/service_context.go`，跟已有的无下划线版本（`performancemiddleware.go`、`servicecontext.go`）冲突。**根因**：这两类文件是手写的（不是靠 goctl `api go` 生成的），本来就没有下划线版本，goctl 加了 `-style go_zero` 之后反而"找不到"这些手写文件，又生成了一份骨架桩。**正确做法**：这类手写文件本来就不该被 goctl 覆盖或旁生，`generate-api.sh` 跑完之后一定要看一眼 `git status` 里 `internal/middleware/`、`internal/svc/` 有没有意外多出来的文件，多出来的直接删（都是空壳桩，不会用到）。
> - **现象**：GGA（Gentleman Guardian Angel，pre-commit 自动 code review hook）第一次 commit 时报了 4 个问题：魔法数字没用 `internal/consts`、新接口没配 RBAC 权限 SQL、gateway 全局 30s 超时和 Ollama 客户端 60s 超时冲突、没补 changelog。**根因**：赶进度先把功能写完，没有对照 `00-workflow.md` 的"标准开发流程"逐条自查就想提交。**正确做法**：新增任何走 Auth+Permission 中间件的接口，字典枚举值必查 `internal/consts` 有没有现成常量，权限初始化 SQL（哪怕没有前端页面、不建菜单）跟接口代码一起写，别等 review 兜底。
