**踩过的坑 —— 绝对不要再犯**：

> - **现象**：本机 `mysql -uroot` 不加 `-h` 参数时报错连接的是 `mysql4.sqlpub.com:3306`，而不是本地或 127.0.0.1。**根因**：本机 `~/.my.cnf`（或类似的 mysql 客户端默认配置）把默认 host 设成了 `oldbai` 库的真实外部地址，这是为了方便日常用 `mysql` CLI 直接连生产旁路库调试，但也意味着任何没显式指定 `-h127.0.0.1` 的 `mysql` 命令都有连到真实外部库的风险。**正确做法**：AI 在这类环境里要验证 SQL 语法/效果时，必须先 `mysql --print-defaults` 或类似方式确认默认连接目标，不能假设"不加 -h 就是连本地"；本次因为不确定本地 MySQL 的 root 密码，也不愿意反复尝试猜测凭证，选择放弃真实连库验证、只做静态审查，这个判断在拿不准数据库真实身份的场景下是对的，应作为默认策略。
> - **现象**：`iam/metric/init_metric.sql` 里种了一整套 `metric:list/create/update/delete` 权限和 `/api/v1/metrics[/:id]` 接口，但从未在 `admin.api`/Go 代码里存在过对应实现，是纯粹的死数据，且靠人工通读没有第一时间发现。**根因**：这类"种子数据 vs. 真实 `.api`/代码"的一致性，光读 SQL 文件本身看不出问题，必须反查 `admin-server/api/admin.api` 和 Go 符号引用才能确认；本次是靠 pre-commit AI 审查触发反查才发现的。**正确做法**：以后凡是修改涉及 `admin_api`/`admin_permission` 种子数据的文件，应主动 `grep` 一下 `admin.api` 和相关 Go 目录确认路由/符号真实存在，不要假设"文件已经在仓库里很久就是对的"。
