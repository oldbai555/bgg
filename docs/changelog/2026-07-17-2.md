**踩过的坑 —— 绝对不要再犯**：

> - **现象**：`deploy/ghcr-dev-mixed` 分支 push 后，GitHub 网页 compare/PR 页面报 `There isn't anything to compare. main and deploy/ghcr-dev-mixed are entirely different commit histories.`，diff 区域只显示分支最后一个 commit 的改动（2 个 `.engram/` 文件），完全看不到真正的 42 个文件改动，也没有 "Create pull request" 按钮。**根因**：`deploy/ghcr-dev-mixed` 是从 `feature/bggadmin` 切出来的，而 `feature/bggadmin` 早先被 squash merge 到 `main` 时就已经和 `main` 断了共同祖先（`git merge-base` 返回空）——squash merge 会在目标分支上产生一份全新提交，不保留与源分支的父子关系，历史断裂但内容其实是同步的。GitHub 网页版 PR 比较用的是三点 diff（依赖 merge-base 计算"分叉以来各自的改动"），没有共同祖先时这个计算直接失效，页面的 fallback 展示具有误导性（不能理解为"这两个分支内容差不多"）。**正确做法**：`git merge-base origin/main <branch>` 判断是否断链；确认真实改动范围要用本地两点 diff `git diff origin/main..<branch> --stat`（snapshot 对比，不依赖祖先）；修复用 `git rebase --onto origin/main <断链点commit> <branch>` 把断链分支的提交重新接到 `origin/main` 上（内容一致的话不会冲突），再 `git push --force-with-lease` 覆盖远端分支，之后 `merge-base` 恢复正常、GitHub 页面变回 "N files changed"、PR 创建流程恢复正常。往后用新增的 `script/branch_from_main.sh` 直接从 `origin/main` 切分支不会再触发这个问题，只有从 `feature/bggadmin` 这类已经断链的历史分支继续派生新分支时才会复现。完整记录也存了一份 Engram（`discovery/squash-merge-github-pr-rebase-onto`）。
> - **现象**：给 `docker-compose.dev-mixed.yml`/`deploy-dev.sh` 加 ghcr 镜像相关注释时，第一版写"TAG 不设默认值"，但 `deploy-dev.sh` 脚本本身其实给 TAG 设了默认值（pull main 后自动取 HEAD sha）。**根因**：写文档和写脚本是分两步做的，写完文档时没有回头核对脚本的真实行为，本仓库的 Gentleman Guardian Angel pre-commit 审查两次都抓到了这个自相矛盾。**正确做法**：注释/规则文档里凡是描述"某个变量是否有默认值/是否必填"这类行为性断言，提交前要回读一遍对应脚本的真实逻辑，不能凭写文档时的记忆。
