**踩过的坑 —— 绝对不要再犯**：

> - **现象**：gateway 部署后连续报 `JWT_ACCESS_SECRET`/`IAM_RPC_ENDPOINT` 等 fail-fast 检查失败。**根因**：服务器仍在按重构前的单体架构用 `supervisor` 部署，`script/admin.sh` 的 `supervisor` 系列命令没跟上 Phase 2 拆分，只打了 gateway 一个包。**正确做法**：拆分架构上线必须用 docker-compose 路径（见 `.cursor/rules/00-workflow.mdc`「部署路径」一节），不能沿用老的 supervisor 单二进制流程。
> - **现象**：飞书登录报"查询第三方账号绑定失败"。**根因**：`admin_user_third_party`、`demo` 两张表在外部库 `oldbai` 里缺失。**正确做法**：新环境首次部署前，先核对目标库的表结构是否和代码依赖的全部一致，不能假设"能建的表都建过了"。
> - **现象**：飞书登录成功但用户部门列显示"-"。**根因**：`add_feishu_pending_department_20260716.sql`/`add_feishu_role_20260716.sql` 两个当天早些时候提交的迁移在 `oldbai` 里没跑过。**正确做法**：迁移 SQL 提交后要单独跟踪是否已在目标库执行，不能假设"提交了就等于跑过了"。
> - **现象**：登录后前端控制台刷屏 24 条 `[Router] 路由注册失败`。**根因**：`admin_menu.component` 字段还是前端 Phase 1-2 重构前的旧目录路径（`system/blog/sdk/video/chatroom` 分组），数据滞后于代码（代码已按 `iam/content/chat/monitoring/misc` 9 域重组）。**正确做法**：前端目录结构重组后，要同步检查数据库里存的路径类字段（菜单 `component`），不能只改代码不改数据。
> - **现象**：数据修完后仍有 3 条残留。**根因**：go-zero `CachedConn` 直连 MySQL 改数据不会自动使 Redis 缓存失效。**正确做法**：手动改库后必须清对应 Redis key（本次是 `cache:adminUser:id:*`/`cache:user:permissions:*`/`cache:user:menu_tree:*`/`cache:menu:tree`），不能只改 MySQL 就认为生效。
> - **现象**：`admin.go` 的 `SyncApiRoutes` 调用改用 `zrpc.WithCallTimeout(30s)` 后，服务器实测仍卡在约 2s。**根因**：怀疑瓶颈在服务端 `services/iam/etc/iam.yaml` 同样没配 `Timeout`（默认 2000ms），`RpcServerConf` 的 `UnaryTimeoutInterceptor` 会用自己的超时覆盖请求 context。**正确做法**：`WithCallTimeout` 只覆盖客户端，服务端超时要单独用 `MethodTimeouts` 放宽，两端都要配置，光改一端不够（尚未验证修复，留给下一个会话）。
