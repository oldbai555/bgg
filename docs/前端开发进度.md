> **本文档是前端开发的唯一指导文档**，整合了项目架构、技术栈、开发规范、功能进度等所有内容。

## 0. 项目概述与技术栈

### 项目概述
本项目是基于 Vue 3 + TypeScript + Vite 构建的现代化前端管理系统，采用组件化、模块化设计，支持权限管理、主题切换、国际化等功能。

### 技术栈

#### 核心框架
- **Vue 3.4+**: 渐进式 JavaScript 框架，使用 Composition API
- **Vite 5.0+**: 下一代前端构建工具，提供极速的开发体验
- **TypeScript 5.3+**: 类型安全的 JavaScript 超集

#### 状态管理
- **Pinia 2.1+**: Vue 3 官方推荐的状态管理库

#### HTTP 请求
- **Axios 1.6+**: Promise 基础的 HTTP 客户端

#### UI 组件库
- **Element Plus 2.4+**: 企业级 UI 组件库
- **Element Plus Icons**: 图标库

#### 路由管理
- **Vue Router 4.2+**: Vue 官方路由库

#### 国际化
- **vue-i18n 9.9+**: Vue 国际化解决方案

#### 其他工具库
- **video.js 8.23+**: 视频播放器（用于视频管理功能）

### 核心架构设计

#### 技术分层
```
┌──────────────────────────────────────┐
│         Page / View Layer            │  (views/*.vue)
├──────────────────────────────────────┤
│      Component Layer                 │  (components/*.vue)
├──────────────────────────────────────┤
│   State Management (Pinia)           │  (stores/*.ts)
├──────────────────────────────────────┤
│   API / Service Layer                │  (api/*.ts)
├──────────────────────────────────────┤
│   HTTP Client (Axios)                │  (utils/request.ts)
├──────────────────────────────────────┤
│   Backend API                        │  (go-zero 后端)
└──────────────────────────────────────┘
```

#### 前后端协同（goctl 生成）

**工作流程**：
```
后端定义 .api → goctl api go 生成后端 → goctl api ts 生成前端 → 前端在 api/ 手动二次封装 → 页面/业务使用
```

**目录约定**：
- `src/api/generated/`：goctl 生成的接口函数（**禁止手改**）
- `src/api/*.ts`：对 generated 代码的二次封装、错误处理、适配
- `admin-server/scripts/generate-ts.sh`：批量生成脚本

**生成步骤**：
1. 安装 goctl：`go install github.com/zeromicro/go-zero/tools/goctl@latest`
2. 单文件生成：`cd admin-server && ./scripts/generate-ts.sh admin.api`
3. 生成代码规范：
   - ✅ 必做：使用 goctl 从 .api 生成 TS 代码；生成文件仅存放 `generated/`
   - ✅ 必做：封装层统一错误处理；遵守 Page → Component → Store → API 分层
   - ❌ 禁止：手动编辑 `generated/` 下的代码；重新定义与 generated 重复的类型

## 1. 开发必读
- 遵循 Page → Component → Store → API → Backend 分层，generated 代码禁止手改。
- goctl 前后端协同：使用 `pnpm api:gen` 或直接执行 goctl 生成 TS（输出到 `src/api/generated/`），业务封装写在 `src/api/*.ts`。
- **API 时间字段规范（强制）**：
  - 后端统一返回 `int64` 类型的时间戳（秒级），不做格式化
  - 前端负责时间格式化显示，使用统一的工具函数（如 `formatTime`, `formatDateTime` 等）将时间戳转换为可读的日期时间字符串
  - 所有涉及时间字段（`createdAt`, `updatedAt`, `deletedAt`, `publishTime`, `readAt` 等）的 API 响应都是 `number` 类型（对应后端的 `int64`）
- **开发流程规范（强制遵守）**：
  1. **确定要开发的功能**：明确功能需求，确定模块名称和功能描述
  2. **生成前端页面骨架**：使用 `scripts/generate-sql.sh -group <group> -name <name>` 生成前端页面 xxx.vue 骨架文件
     - **脚本执行规范**：如果需要执行 `scripts/` 目录下的脚本，需要用户执行后再通知进行下一步；如果是其他脚本（如数据库 SQL 脚本），AI 可以直接执行
  3. **等待后端完成**：等待后端完成 API 定义和实现，确保后端接口可独立测试通过
  4. **生成 TS 代码**：使用 `scripts/generate-ts.sh <api_file>` 从后端 .api 文件生成 TS 代码
     - **脚本执行规范**：如果需要执行 `scripts/` 目录下的脚本，需要用户执行后再通知进行下一步；如果是其他脚本（如数据库 SQL 脚本），AI 可以直接执行
  5. **完善前端页面**：基于生成的页面骨架和 TS 代码，完善前端页面和组件功能
  6. **前后端联调**：前后端联调接口，测试完整流程，确保功能正常
  7. **测试通过后再开发下一个功能**：每个功能必须完整实现并测试通过后，才能开始下一个功能的开发
- **通用组件使用规范**：
  - **D2Table 组件**（`src/components/common/D2Table.vue`）：适用于所有涉及表格展示、分页、详情/编辑、新增的业务页面
    - 功能：统一的表格展示、分页、内置详情/编辑抽屉、内置新增抽屉、支持多种列类型
    - 使用场景：列表管理页面（如角色管理、权限管理、用户管理等）
    - 参考示例：`src/views/system/RoleList.vue`、`src/views/system/PermissionList.vue`、`src/views/system/UserList.vue`
    - 组件文档：`src/components/common/README.md`
    - 类型定义：`src/types/table.ts`
  - **el-tree 组件**：适用于树形数据展示（如部门管理、菜单管理）
    - 参考示例：`src/views/system/DepartmentList.vue`、`src/views/system/MenuList.vue`
- **数据字典使用规范（必须遵守）**：
  - **字典的使用场景**：
    - **状态枚举**：用户状态、订单状态、审核状态等需要在多个地方使用的状态值
    - **类型分类**：文件类型、存储类型、支付方式、物流方式等业务类型
    - **选项列表**：性别、是否、地区、行业等需要在表单中下拉选择的选项
    - **业务常量**：需要在多个模块中复用的常量值，如用户等级、会员类型等
    - **配置选项**：需要动态调整且可能频繁变更的配置项，如通知类型、消息模板类型等
  - **优先考虑使用字典的情况**：
    - 前端需要在下拉选择框（`el-select`）、单选框（`el-radio-group`）、多选框（`el-checkbox-group`）等组件中展示的选项
    - 需要在多个页面或模块中复用的枚举值
    - 需要由业务人员或管理员动态维护的选项（不需要改代码）
    - 需要支持多语言或国际化显示的选项
    - 需要在表格列中显示标签（`el-tag`）的状态或类型字段
    - 需要在搜索条件中使用的筛选选项
  - **字典使用流程**：
    1. 在开发新功能时，先评估字段是否需要使用字典
    2. 如果需要，**后端开发人员编写增量字典插入 SQL 语句**（见后端文档「字典 SQL 插入规范」）
    3. 后端执行字典 SQL 语句，创建字典类型和字典项
    4. **前端统一字典管理**：在 `src/stores/dict.ts` 的 `REQUIRED_DICT_CODES` 数组中添加新的字典类型编码
    5. **登录后自动加载**：用户登录后会自动批量加载所有必需的字典数据（通过 `POST /api/v1/dict/batch` 接口）
    6. **在组件中使用字典**：使用 `useDictOptions` composable 获取字典选项，避免硬编码
    7. 在表格列中使用字典项进行值转换显示（如状态标签）
  - **字典 SQL 插入规范（后端负责）**：
    - **字典数据通过 SQL 语句新增**：所有字典类型和字典项的数据都通过 SQL 语句插入，不在管理界面中手动创建
    - **增量 SQL 格式**：在 `admin-server/db/data.sql` 文件中，字典相关的 SQL 应放在统一位置
    - **字典类型 SQL 模板**：
      ```sql
      -- {功能模块}相关字典类型
      INSERT INTO `admin_dict_type` (`id`, `name`, `code`, `description`, `status`, `created_at`, `updated_at`, `deleted_at`)
      VALUES 
        ({id}, '{字典类型名称}', '{字典类型编码}', '{字典类型描述}', 1, UNIX_TIMESTAMP(), UNIX_TIMESTAMP(), 0)
      ON DUPLICATE KEY UPDATE 
        `name`=VALUES(`name`), 
        `description`=VALUES(`description`), 
        `updated_at`=UNIX_TIMESTAMP(), 
        `deleted_at`=0;
      ```
    - **字典项 SQL 模板**：
      ```sql
      -- {字典类型名称}字典项
      INSERT INTO `admin_dict_item` (`id`, `type_id`, `label`, `value`, `sort`, `status`, `remark`, `created_at`, `updated_at`, `deleted_at`)
      VALUES 
        ({id1}, {type_id}, '{字典项标签1}', '{字典项值1}', 1, 1, '{备注1}', UNIX_TIMESTAMP(), UNIX_TIMESTAMP(), 0),
        ({id2}, {type_id}, '{字典项标签2}', '{字典项值2}', 2, 1, '{备注2}', UNIX_TIMESTAMP(), UNIX_TIMESTAMP(), 0)
      ON DUPLICATE KEY UPDATE 
        `label`=VALUES(`label`), 
        `value`=VALUES(`value`), 
        `sort`=VALUES(`sort`), 
        `status`=VALUES(`status`), 
        `remark`=VALUES(`remark`), 
        `updated_at`=UNIX_TIMESTAMP(), 
        `deleted_at`=0;
      ```
    - **前端使用字典的注意事项**：
      - 前端不需要关心字典数据的创建，只需通过 API 获取字典项列表
      - 如果字典数据不存在，API 会返回空列表，前端应做容错处理
      - 字典项的 `value` 字段通常存储为字符串，前端使用时可能需要类型转换
  - **前端统一字典管理（必须遵守）**：
    - **字典 Store**：`src/stores/dict.ts` 统一管理全局字典数据
    - **批量加载**：登录后通过 `POST /api/v1/dict/batch` 接口一次性加载所有必需的字典
    - **必需字典列表**：在 `REQUIRED_DICT_CODES` 数组中维护所有需要加载的字典类型编码
    - **自动加载时机**：用户登录成功后自动调用 `dictStore.loadDicts()` 加载字典
    - **字典数据缓存**：字典数据存储在 Pinia store 中，整个会话期间有效，登出时清除
  - **前端字典使用方式（推荐使用 useDictOptions）**：
    ```typescript
    // ✅ 推荐：使用 useDictOptions composable（响应式，自动从store获取）
    import { useDictOptions } from '@/composables/useDictOptions';
    
    // 在组件中使用
    const { options: statusOptions, getLabel: getStatusLabel } = useDictOptions(
      'user_status',
      [
        { label: '启用', value: '1' },  // 默认值（字典为空时使用）
        { label: '禁用', value: '0' }
      ]
    );
    
    // 在模板中使用
    <el-select v-model="query.status">
      <el-option
        v-for="item in statusOptions"
        :key="item.value"
        :label="item.label"
        :value="item.value"
      />
    </el-select>
    
    // 在 D2Table 列配置中使用
    const drawerColumns = computed<DrawerColumn[]>(() => [
      {
        prop: 'status',
        label: '状态',
        type: D2TableElemType.Select,
        options: statusOptions.value  // 注意：需要 .value 获取实际数组
      }
    ]);
    ```
  - **字典使用禁止事项**：
    - ❌ **禁止硬编码选项**：所有 `el-select`、`el-radio-group`、`el-checkbox-group` 等组件的选项必须从字典获取
    - ❌ **禁止直接调用 dictGet API**：应使用 `useDictOptions` composable 或 `dictStore.getDictOptions()`
    - ❌ **禁止在组件中单独加载字典**：字典应在登录后统一批量加载，不要在组件中单独调用 API
    - ✅ **必须添加到 REQUIRED_DICT_CODES**：新增字典类型后，必须在 `src/stores/dict.ts` 的 `REQUIRED_DICT_CODES` 数组中添加

## 2. 核心功能实现

### 2.1 认证与登录
- **登录页面**：`src/views/Login.vue`

- **路由级权限**：`src/router/index.ts`（meta.permission）

### 2.3 菜单管理
- **菜单数据**：`src/stores/user.ts`（menus 数组，从 `/api/v1/menus/my-tree` 获取）

- **动态路由**：`src/router/index.ts`（根据菜单树自动注入路由）
- **菜单缓存**：localStorage 缓存 5 分钟（TTL）

### 2.4 数据字典
- **字典 Store**：`src/stores/dict.ts`（统一管理全局字典数据）
- **批量加载**：登录后通过 `POST /api/v1/dict/batch` 一次性加载所有必需的字典
- **必需字典列表**：在 `REQUIRED_DICT_CODES` 数组中维护
- **字典使用**：`src/composables/useDictOptions.ts`（useDictOptions composable）

### 2.5 通用组件
- **D2Table 组件**：`src/components/common/D2Table.vue`
  - 功能：统一的表格展示、分页、内置详情/编辑抽屉、内置新增抽屉
  - 使用场景：所有列表管理页面（如角色管理、权限管理、用户管理等）
  - 参考示例：`src/views/system/RoleList.vue`、`src/views/system/UserList.vue`

## 3. 开发规范

### 3.1 代码规范
- **ESLint + Prettier**：统一代码风格
- **TypeScript 严格模式**：类型安全
- **组件命名**：PascalCase（如 `UserList.vue`）
- **文件命名**：kebab-case（如 `use-dict-options.ts`，但 composables 使用 camelCase）

### 3.2 API 调用规范
- **统一从 `@/api/generated/admin` 导入**：使用 goctl 生成的接口函数
- **二次封装**：在 `src/api/*.ts` 中统一错误处理、适配返回结构
- **时间字段规范**：后端返回 `int64` 时间戳（秒级），前端负责格式化显示

### 3.3 字典使用规范
- **禁止硬编码选项**：所有下拉选择、单选框等组件的选项必须从字典获取
- **使用 useDictOptions**：统一使用 `useDictOptions` composable 获取字典选项

  - 导出接口调用成功后，通过右下角浮动任务球或「任务管理」页面查看任务状态；

  - 由文件管理模块暴露的 URL 统一承载下载逻辑，前端不直接拼接物理路径，也不绕开文件管理模块。

## 10. Public 界面开发规范（小程序风格）

### 10.1 规范概述

Public 界面（`src/views/public/*`）采用统一的「小程序风格」设计规范，确保所有公共页面（博客、视频等）具有一致的视觉体验和交互模式。

**核心设计原则：**
- **统一背景**：整页使用柔和暖色渐变背景 `linear-gradient(135deg, #fff7e6 0%, #ffe9d9 45%, #ffd1a4 100%)`，内容区域为白色卡片
- **扁平化设计**：轻微阴影（`box-shadow: 0 2px 8px rgba(0, 0, 0, 0.05)`），避免过度装饰
- **响应式布局**：PC 端多列网格，移动端（`≤768px`）单列信息流
- **统一分页**：移动端简化分页组件，只保留「上一页 / 页码 / 下一页」
