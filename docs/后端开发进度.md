> **本文档是后端开发的唯一指导文档**，整合了项目架构、技术栈、开发规范、功能进度等所有内容。

## 0. 项目概述与技术栈

### 项目概述
本项目是基于 go-zero 框架构建的现代化后端管理系统，采用分层架构设计，支持 RBAC 权限管理、系统配置、数据字典、文件存储、日志监控等功能。

### 技术栈

#### 核心框架
- **go-zero 1.9.3+**: 微服务框架，高性能、易扩展
- **Go 1.24+**: 编程语言

#### 数据访问
- **go-zero sqlx + cache**: 数据访问层，支持自动缓存
- **MySQL 8.0+**: 关系型数据库
- **Redis 6.0+**: 缓存与会话存储
- **squirrel**: SQL 构建库（用于动态 SQL 构建）

#### 认证授权
- **JWT (golang-jwt/jwt/v4)**: 令牌认证（双令牌方案：Access Token + Refresh Token）
- **bcrypt (golang.org/x/crypto)**: 密码加密

#### 日志与监控
- **go-zero logx**: 内置日志库
- **gopsutil**: 系统监控指标采集

#### 其他工具
- **goctl**: go-zero 代码生成工具
- **uuid (google/uuid)**: UUID 生成
- **gorilla/websocket**: WebSocket 支持

### 核心架构设计

#### 分层架构
```
┌─────────────────────────────────────┐
│         API Layer (HTTP/RPC)        │  (api/*.api, handler/*)
├─────────────────────────────────────┤
│         Logic Layer                 │  (logic/*)
├─────────────────────────────────────┤
│         Repository Layer            │  (repository/*)
├─────────────────────────────────────┤
│         Model/Entity Layer          │  (model/*, goctl 生成)
├─────────────────────────────────────┤
│         Database Layer (MySQL)      │  (db/*.sql)
└─────────────────────────────────────┘
```

> **数据访问层统一使用 goctl model 生成的 sqlx + cache 代码**，不再保留手写 GORM 版本；Model 生成使用自定义模板，内置 `created_at/updated_at/deleted_at`、软删除、分页与分片查询。

#### 中间件体系
1. **PerformanceMiddleware**: 性能监控中间件（记录响应时间、慢接口）
2. **RateLimitMiddleware**: 限流中间件（按 IP、用户、接口限流）
3. **AuthMiddleware**: JWT 鉴权中间件（校验 Access Token + 黑名单）
4. **PermissionMiddleware**: 权限验证中间件（基于用户权限验证 API 访问）
5. **OperationLogMiddleware**: 操作日志中间件（自动记录增删改操作）
6. **ApiEnabledMiddleware**: API 启用控制中间件（业务级开关）
7. **SDKAuthMiddleware**: SDK 鉴权中间件（key+secret 鉴权）
8. **SDKRateLimitMiddleware**: SDK 限流中间件（按 key+apiCode 限流）
9. **SDKCallLogMiddleware**: SDK 调用记录中间件（记录 SDK 调用日志）

#### 前后端协同（goctl 生成）

**工作流程**：
```
后端定义 .api → goctl api go 生成后端 → goctl api ts 生成前端 → 前端在 api/ 手动二次封装 → 页面/业务使用
```

**代码生成规范**：
- **Model 代码生成**：使用 `goctl model mysql ddl` 从 SQL 文件生成 Model 代码
  - 推荐使用脚本：`./scripts/generate-model.sh db/tables.sql`
  - 使用项目自定义模板（`admin-server/.template`），支持统一时间戳字段、软删除、分页/分片查询
- **API Handler 代码生成**：使用 `goctl api go` 从 `.api` 文件生成 Handler/Logic 骨架
  - 推荐使用脚本：`./scripts/generate-api.sh api/admin.api`
- **前端 TypeScript 代码生成**：使用 `goctl api ts` 从 `.api` 文件生成前端类型和 API 代码
  - 推荐使用脚本：`./scripts/generate-ts.sh api/admin.api`
- **原则**：能使用 go-zero 工具生成的代码，一律使用工具生成；只有在业务需要特殊处理时，才进行自定义开发

## 1. 开发必读
- 按文档分层（API → Handler → Logic → Repository → Model），禁止保留旧兼容层。
- 完成功能记入「已完成功能」，分阶段需求放「待实现 / 待完善功能」，技术约束写入「技术决策记录」。
- 统一响应/错误码、JWT 双令牌、bcrypt 密码、logx 日志、go-zero sqlx + cache 数据访问、go-zero 限流。
- **开发流程规范（强制遵守）**：
  1. **确定要开发的功能**：明确功能需求，确定模块名称和功能描述
  2. **生成初始化 SQL**：使用 `scripts/generate-sql.sh -group <group> -name <name>` 生成初始化表 SQL 语句，以及对应的权限菜单接口等 SQL 语句、前端页面 xxx.vue、xxx.api 文件
     - **脚本执行规范**：如果需要执行 `scripts/` 目录下的脚本，需要用户执行后再通知进行下一步；如果是其他脚本（如数据库 SQL 脚本），AI 可以直接执行
  3. **补齐初始化表 SQL**：检查生成的 SQL，补齐初始化表 SQL 语句所需要的字段（如 created_at、updated_at、deleted_at 等）
  4. **补齐 CRUD 接口参数**：检查生成的 .api 文件，补齐 CRUD 接口的参数定义
  5. **生成 Model 代码**：使用 `scripts/generate-model.sh <sql_file>` 生成对应的 Model 代码
     - **脚本执行规范**：如果需要执行 `scripts/` 目录下的脚本，需要用户执行后再通知进行下一步；如果是其他脚本（如数据库 SQL 脚本），AI 可以直接执行
  6. **生成 API 代码**：使用 `scripts/generate-api.sh <api_file>` 生成对应的 Handler/Logic 代码骨架
     - **脚本执行规范**：如果需要执行 `scripts/` 目录下的脚本，需要用户执行后再通知进行下一步；如果是其他脚本（如数据库 SQL 脚本），AI 可以直接执行
  7. **开发功能**：实现 Repository、Logic、Handler 的业务逻辑，完成后执行对应的 SQL 语句，创表和写入对应的权限菜单等
  8. **启动后端服务**：确保后端服务能正常启动，接口可独立测试通过
  9. **完成前端页面**：将生成的前端 xxx.vue 进行页面完善，然后能正常启动前端项目，进行前后端联调
  10. **测试通过后再开发下一个功能**：每个功能必须完整实现并测试通过后，才能开始下一个功能的开发
- **权限 SQL 生成规范（必须遵守）**：
  - 每新增一个功能模块，必须同时生成对应的权限列表 SQL 文件
  - SQL 文件命名：`admin-server/db/permissions_{module_name}.sql`
  - 权限编码规范：`{module}:{action}`（如 `user:list`、`user:create`、`user:update`、`user:delete`）
  - 权限 SQL 必须包含：列表、新增、编辑、删除四个基础权限
  - 执行权限 SQL 后，需要在菜单管理中添加对应的菜单项，并关联权限编码
  - 参考示例：`admin-server/db/permissions_menu_user.sql`
- **数据字典使用规范（必须遵守）**：
  - **字典的使用场景**：
    - **状态枚举**：用户状态、订单状态、审核状态等需要在多个地方使用的状态值
    - **类型分类**：文件类型、存储类型、支付方式、物流方式等业务类型
    - **选项列表**：性别、是否、地区、行业等需要在表单中下拉选择的选项
    - **业务常量**：需要在多个模块中复用的常量值，如用户等级、会员类型等
    - **配置选项**：需要动态调整且可能频繁变更的配置项，如通知类型、消息模板类型等
  - **优先考虑使用字典的情况**：
    - 前端需要在下拉选择框、单选框、多选框等组件中展示的选项
    - 需要在多个页面或模块中复用的枚举值
    - 需要由业务人员或管理员动态维护的选项（不需要改代码）
    - 需要支持多语言或国际化显示的选项
    - 需要按业务规则动态调整的选项（如根据权限、地区等过滤）
    - 需要在报表、统计、导出等功能中使用的分类字段
  - **不建议使用字典的情况**：
    - 系统级别的固定枚举（如 HTTP 方法、数据库类型、通用状态 `"ok"/"error"`、Redis Key 前缀、固定路径 `/api/v1/ping`、限流提示文案等），应使用代码常量，并统一放在 `admin-server/internal/consts` 包中集中维护。
    - 只在单一模块内部使用的简单枚举，可在模块内部使用局部常量，但禁止在业务逻辑中直接硬编码字符串（推荐在对应模块的 `const` 块集中定义）。
    - 需要复杂业务逻辑判断的选项，应使用专门的业务表。
- **字典使用流程**：
  1. 在开发新功能时，先评估字段是否需要使用字典
  2. 如果需要，**编写增量字典插入 SQL 语句**（见下方「字典 SQL 插入规范」）
  3. 将字典 SQL 语句添加到 `admin-server/db/data.sql` 文件中，或创建独立的字典初始化 SQL 文件
  4. 执行 SQL 语句，创建字典类型和字典项
  5. 前端通过 `/api/v1/dict?code={dict_code}` 接口获取字典项列表
  6. 后端在 Logic 层可以通过 Repository 查询字典项进行验证或转换
- **API 时间字段规范（强制）**：
  - **统一规则**：所有涉及时间字段（`createdAt`, `updatedAt`, `deletedAt`, `publishTime`, `readAt`, `loginAt`, `logoutAt` 等）的 API 响应，后端统一返回 `int64` 类型的时间戳（秒级），**不做任何格式化**
  - **后端实现**：Logic 层直接返回数据库中的 `int64` 时间戳，禁止使用 `time.Format()` 或 `strconv.FormatInt()` 进行格式化
  - **前端处理**：前端负责时间格式化显示，使用统一的工具函数（如 `formatTime`, `formatDateTime` 等）将时间戳转换为可读的日期时间字符串
  - **示例**：
    ```go
    // ❌ 错误：后端格式化时间
    CreatedAt: time.Unix(log.CreatedAt, 0).Format("2006-01-02 15:04:05")
    
    // ✅ 正确：直接返回时间戳
    CreatedAt: log.CreatedAt
    ```
  - **字典 SQL 插入规范（必须遵守）**：
    - **字典数据通过 SQL 语句新增**：所有字典类型和字典项的数据都通过 SQL 语句插入，不在管理界面中手动创建
    - **增量 SQL 格式**：在 `admin-server/db/data.sql` 文件中，字典相关的 SQL 应放在统一位置，使用注释标识
    - **字典类型 SQL 模板**：
      ```sql
      -- {功能模块}相关字典类型
      INSERT INTO `admin_dict_type` (`id`, `name`, `code`, `description`, `status`, `created_at`, `updated_at`, `deleted_at`)
      VALUES 
        ({id}, '{字典类型名称}', '{字典类型编码}', '{字典类型描述}', 1, UNIX_TIMESTAMP(), UNIX_TIMESTAMP(), 0)
      ON DUPLICATE KEY UPDATE 
        `name`=VALUES(`name`), 
        `description`=VALUES(`description`), 
        `updated_at`=UNIX_TIMESTAMP(), 
        `deleted_at`=0;
      ```
    - **字典项 SQL 模板**：
      ```sql
      -- {字典类型名称}字典项
      INSERT INTO `admin_dict_item` (`id`, `type_id`, `label`, `value`, `sort`, `status`, `remark`, `created_at`, `updated_at`, `deleted_at`)
      VALUES 
        ({id1}, {type_id}, '{字典项标签1}', '{字典项值1}', 1, 1, '{备注1}', UNIX_TIMESTAMP(), UNIX_TIMESTAMP(), 0),
        ({id2}, {type_id}, '{字典项标签2}', '{字典项值2}', 2, 1, '{备注2}', UNIX_TIMESTAMP(), UNIX_TIMESTAMP(), 0)
      ON DUPLICATE KEY UPDATE 
        `label`=VALUES(`label`), 
        `value`=VALUES(`value`), 
        `sort`=VALUES(`sort`), 
        `status`=VALUES(`status`), 
        `remark`=VALUES(`remark`), 
        `updated_at`=UNIX_TIMESTAMP(), 
        `deleted_at`=0;
      ```
    - **ID 分配规范**：
      - 字典类型 ID：从 1 开始连续递增，参考 `admin-server/db/data.sql` 中已有的最大 ID
      - 字典项 ID：从 1 开始连续递增，参考 `admin-server/db/data.sql` 中已有的最大 ID
      - 确保 ID 不冲突，建议在现有最大 ID 基础上递增
    - **SQL 执行时机**：
      - 在开发新功能时，确定需要使用的字典后，立即编写并执行字典 SQL 语句
      - 字典 SQL 应在创建业务表之前或同时执行，确保字典数据可用
      - 执行后验证字典数据是否正确插入，可通过字典管理界面或直接查询数据库验证

## 2. 核心功能实现

### 2.1 认证与授权
- **登录流程**：用户名/密码验证 → bcrypt 密码比对 → 生成 JWT 双令牌（Access Token + Refresh Token）
- **Token 管理**：Access Token（短期，如 2 小时）+ Refresh Token（长期，如 7 天）
- **黑名单机制**：登出时将 Token 加入 Redis 黑名单（`jwt:blacklist:*`）
- **权限验证**：基于 RBAC 模型，支持用户-角色-权限三层关联

### 2.2 RBAC 权限管理
- **用户管理**：用户 CRUD、状态管理、角色分配
- **角色管理**：角色 CRUD、权限分配
- **权限管理**：权限 CRUD、菜单/接口关联
- **部门管理**：部门树形结构、部门成员管理
- **菜单管理**：菜单树形结构、按钮级权限（type=3）
- **接口管理**：API 资源定义、权限关联

### 2.3 系统支撑
- **系统配置**：配置项管理、缓存策略、热更新
- **数据字典**：字典类型/项管理、批量查询、缓存优化
- **文件管理**：文件上传/下载、本地存储、文件记录

### 2.4 日志与监控
- **操作日志**：自动记录增删改操作、异步写入、支持导出
- **登录日志**：记录登录/登出信息、IP/设备信息、统计功能
- **审计日志**：记录敏感操作（权限变更、配置修改等）
- **性能日志**：接口响应时间监控、慢接口标记
- **系统监控**：CPU/内存/磁盘/网络监控、系统统计
- **通用打点统计（PV/UV/VV/IP）**：
  - 上报接口：`POST /api/v1/metrics/report`（模块：`metric`），支持 `blog_article_list/blog_article_detail/video_list/video_detail` 等 module，并在 Logic 层对 `module` 做白名单校验（常量集中在 `internal/consts/blog.go` 等处）；
  - 实时统计：使用 Redis key `metric:{module}:{bizId}:{day}:{pv|uv|vv|ip}` 做当日实时统计（PV/UV/VV/IP 计数）；
  - 日汇总表：MySQL 表 `metric_daily_stats`（字段：module/biz_id/day/pv/uv/vv/ip + created_at/updated_at/deleted_at），`MetricReportLogic` 在完成 Redis 自增后使用独立 `context.Background()` + 超时控制异步调用 `MetricRepository.UpsertDailyStats` 做增量落库，避免 HTTP 请求上下文取消导致的 `context canceled`；
  - 管理后台查询接口：`GET /api/v1/metrics/stats`（模块：`metric_admin`），通过 `MetricStatsLogic` 从 `metric_daily_stats` 读取单日聚合结果：`bizId>0` 时按 module+bizId+day 精确查询，`bizId=0` 时按 module+day 维度做 SUM 聚合返回整体 PV/UV/VV/IP，用于后台“数据统计”页面。

### 2.5 缓存策略
- **go-zero SQL 查询缓存**：自动缓存单条记录查询（FindOne 等）
- **业务层缓存**：菜单树、权限列表、字典项、配置项缓存
- **缓存过期时间**：用户权限（30分钟）、菜单树（30分钟）、字典（1小时）、配置（10分钟）
- **缓存更新策略**：数据变更时主动清除相关缓存

### 2.6 SDK 管理
- **API Key 管理**：key+secret 生成、状态/过期/IP 白名单管理
- **接口管理**：接口定义、默认限频、状态管理
- **授权关系**：key-接口授权、自定义限频
- **调用记录**：请求/响应记录、耗时统计、导出功能
- **SDK 中间件**：鉴权、限流、调用记录

## 3. 开发规范

### 3.1 代码规范
- **分层架构**：严格遵循 Handler → Logic → Repository → Model 分层
- **错误处理**：统一使用 `pkg/errs` 错误码和错误封装
- **响应格式**：统一使用 `pkg/response` 响应封装
- **常量管理**：系统级固定枚举统一放入 `internal/consts` 包

### 3.2 API 设计规范
- **RESTful 设计**：遵循 RESTful 规范
- **统一响应格式**：`{code, msg, data}` 结构
- **时间字段规范**：所有时间字段统一返回 `int64` 类型的时间戳（秒级），不做格式化
- **路径参数**：禁止使用路径参数 `:id`，统一使用 Query/Body 参数

### 3.3 数据库设计规范
- **统一时间戳字段**：所有表必须包含 `created_at`、`updated_at`、`deleted_at` 字段（BIGINT 类型，秒级时间戳，默认值 0）
- **软删除**：通过 `deleted_at` 字段实现软删除（0 表示未删除，>0 表示已软删除）
- **分页查询**：所有列表接口强制分页，默认每页 20 条，最大每页 100 条
- **SQL 构建规范**：所有需要动态构建 SQL 的地方必须使用 `squirrel` 库，禁止使用 `fmt.Sprintf` 或字符串拼接

### 3.4 字典使用规范
- **系统常量**：系统级固定枚举（如 HTTP 方法、通用状态等）统一放入 `internal/consts` 包
- **业务字典**：业务可配置枚举（如订单状态、通知类型等）统一走数据字典方案
- **字典 SQL**：字典数据通过 SQL 语句插入，使用增量 SQL 文件（`db/migrations/dict_*.sql`）

### 3.5 中间件使用规范
- **中间件声明顺序**：必须按执行顺序声明（Performance → RateLimit → Auth → Permission → OperationLog）
- **PermissionMiddleware 与 ApiEnabledMiddleware**：互斥，不能同时存在
- **中间件配置**：在 `api/admin.api` 的 `@server` 中声明

### 3.6 环境配置

#### 开发环境
- **开发服务器**：`go run admin.go` 或使用 Air 热重载
- **配置文件**：`etc/admin-api.yaml`
- **数据库**：MySQL 8.0+
- **缓存**：Redis 6.0+

#### 生产环境
- **构建命令**：`go build -o admin-server admin.go`
- **配置文件**：通过环境变量或配置文件指定
- **部署**：使用 systemd 或 Docker 部署

## 6. 技术决策记录
- 2025-12-23：后台按 go-zero + sqlx + cache 重建，不保留旧兼容路径；目录遵循 Handler → Logic → Repository → Model（使用 go-zero 生成的 logic 层，不再使用 Service 层）。
- 2025-12-26：数据访问层统一使用 goctl 生成的 sqlx + cache Model，不再使用 GORM。
- 2025-12-30：Redis 客户端统一使用 go-zero `stores/redis` 组件，不再直接依赖 go-redis/v9；系统级固定枚举/常量统一放入 `internal/consts` 包，禁止在业务代码中直接硬编码字符串（优先使用常量与数据字典方案）。
- 2026-01-13：**表格导出统一使用异步任务机制**：所有导出类接口（操作日志、登录日志、审计日志、SDK 调用日志、性能日志）一律通过创建 `admin_task` 记录 + `ExcelExportExecutor` 异步生成 CSV，再由文件管理模块创建 `AdminFile` 记录并返回下载 URL；禁止新增直接向 HTTP 响应流同步写出 CSV 的接口，新导出功能必须：
  - 后端：定义导出请求结构体 → 在 Logic 中组装 `ExcelExportParams`（包含 module 与 filters，使用常量 `consts.TaskModuleXXX` 和 `consts.TaskFilterXXX`）→ 通过内部任务创建接口写入 `admin_task`，设置 `type=consts.TaskTypeExcelExport`、`execution_type=consts.TaskExecutionTypeAsync`，并由任务调度器执行；
  - 前端：导出按钮只负责调用导出接口触发任务创建，不直接处理 `blob` 下载；用户在异步任务列表/浮动任务球中查看任务状态与任务结果 JSON 的 `fileUrl`，通过文件管理模块统一下载文件。
 - 2026-01-15：**通用打点模块与统计规范**：
   - Metric 上报接口 `POST /api/v1/metrics/report` 作为通用打点入口，所有业务模块（博客文章列表/详情、视频列表/详情等）的 PV/UV/VV/IP 上报统一复用该接口；
   - 打点模块中的 `module` 参数统一使用常量枚举并在 Logic 层做白名单校验，非法 module 直接返回业务错误码；
   - 实时统计使用 Redis，日汇总使用 MySQL 表 `metric_daily_stats`，通过 `MetricRepository.UpsertDailyStats` 实现 `INSERT ... ON DUPLICATE KEY UPDATE` 增量更新；
   - 异步落库必须使用独立 `context.Background()` + 超时控制，禁止复用 HTTP 请求的 context，避免返回后 goroutine 被取消（`context canceled`）；
   - 后台统计查询统一走 `GET /api/v1/metrics/stats`，不再直接从 Redis 查询，便于统计数据长期留存与离线分析。

## 10. HTTP Range 请求原理与边下边播实现

### 10.1 Range 请求是什么？

HTTP Range 请求是 HTTP/1.1 协议定义的一个功能，允许客户端请求服务器返回资源的**部分内容**，而不是整个文件。这对于视频播放、文件下载续传等场景非常重要。

### 10.2 Range 请求的基本格式

**客户端请求头：**
```
Range: bytes=0-1023
```
表示请求文件的第 0 到 1023 字节（共 1024 字节）。

**服务器响应：**
- 如果支持 Range 请求，返回状态码 `206 Partial Content`
- 响应头包含 `Content-Range: bytes 0-1023/10240`（表示返回 0-1023 字节，文件总大小 10240 字节）
- 响应体只包含请求的字节范围

**完整示例：**
```
请求：
GET /video.ts HTTP/1.1
Range: bytes=0-1023

响应：
HTTP/1.1 206 Partial Content
Content-Range: bytes 0-1023/10240
Content-Length: 1024
Accept-Ranges: bytes

[1024 字节的数据]
```

### 10.3 为什么需要 Range 请求？

#### 传统方式（无 Range 请求）：
1. 客户端请求整个文件
2. 服务器返回完整文件（可能很大，如 100MB）
3. 客户端必须等待完整下载后才能播放
4. **问题**：延迟高、内存占用大、无法拖拽播放

#### Range 请求方式（边下边播）：
1. 客户端请求文件的前 1MB（`Range: bytes=0-1048575`）
2. 服务器只返回这 1MB 数据
3. 客户端收到数据后立即开始播放
4. 播放过程中，客户端继续请求后续片段
5. **优势**：低延迟、低内存占用、支持拖拽播放

### 10.4 视频播放中的 Range 请求

#### M3U8 视频播放流程：

1. **加载 M3U8 文件**：
   ```
   客户端 → 代理服务器 → 源服务器
   GET /playlist.m3u8
   ```
   M3U8 文件很小（几 KB），通常不需要 Range 请求。

2. **解析 M3U8，获取 TS 片段列表**：
   ```m3u8
   #EXTM3U
   #EXTINF:10.0,
   segment001.ts
   segment002.ts
   segment003.ts
   ...
   ```

3. **播放器请求 TS 片段（使用 Range 请求）**：
   ```
   客户端 → 代理服务器 → 源服务器
   GET /segment001.ts
   Range: bytes=0-1023
   
   代理服务器透传 Range 请求头：
   代理服务器 → 源服务器
   GET /segment001.ts
   Range: bytes=0-1023
   
   源服务器响应：
   HTTP/1.1 206 Partial Content
   Content-Range: bytes 0-1023/10240
   [数据流]
   
   代理服务器透传响应：
   代理服务器 → 客户端
   HTTP/1.1 206 Partial Content
   Content-Range: bytes 0-1023/10240
   [数据流]
   ```

4. **边下边播**：
   - 播放器收到第一个片段后立即开始播放
   - 同时继续请求后续片段
   - 实现流畅播放体验

### 10.5 本系统实现方案

#### 后端实现（`m3u8proxylogic.go`）：

1. **透传 Range 请求头**：
   ```go
   // 透传客户端的 Range 请求头到源服务器
   if rangeHeader := r.Header.Get("Range"); rangeHeader != "" {
       httpReq.Header.Set("Range", rangeHeader)
   }
   ```

2. **处理 Range 响应**：
   ```go
   // 支持 200 OK 和 206 Partial Content
   if resp.StatusCode != http.StatusOK && resp.StatusCode != http.StatusPartialContent {
       return err
   }
   
   // 保留 Range 相关响应头
   if lowerKey == "content-range" || lowerKey == "accept-ranges" || lowerKey == "content-length" {
       w.Header().Set(k, v[0])
   }
   ```

3. **流式传输**：
   ```go
   // 使用 io.Copy 直接流式传输，不缓存完整文件
   io.Copy(w, resp.Body)
   ```

4. **CORS 支持**：

   - **前端无需额外配置**

### 10.6 性能优势

1. **低延迟**：
   - 传统方式：需要等待完整文件下载（如 100MB 视频需要 10 秒）
   - Range 方式：只需等待第一个片段（如 1MB 需要 0.1 秒）
   - **延迟降低 100 倍**

2. **低内存占用**：
   - 传统方式：需要缓存完整文件（100MB）
   - Range 方式：只需缓存当前播放片段（1MB）
   - **内存占用降低 100 倍**

3. **支持拖拽播放**：
   - 用户拖拽到 50% 位置
   - 播放器发送 `Range: bytes=50%-100%`
   - 服务器只返回后半部分数据
   - **无需下载前半部分**

4. **网络带宽优化**：
   - 用户可能只看视频的前 10 分钟
   - 使用 Range 请求，只下载需要的部分
   - **节省 90% 带宽**

### 10.7 关键技术点

1. **HTTP 客户端连接池**：
   ```go
   httpClient = &http.Client{
       Transport: &http.Transport{
           MaxIdleConns:        100,
           MaxIdleConnsPerHost: 10,
           IdleConnTimeout:     90 * time.Second,
       },
   }
   ```
   - 复用连接，减少连接建立时间
   - 提升并发性能

2. **流式传输**：
   ```go
   io.Copy(w, resp.Body)
   ```
   - 不等待完整下载
   - 边下载边传输
   - 降低内存占用

3. **响应头透传**：
   - 保留 `Content-Range`、`Content-Length`、`Accept-Ranges`
   - 确保播放器能正确解析 Range 响应

### 10.8 更高级的视频传输方案

除了 HTTP Range 请求，还有以下更高级的方案可以实现视频边下边播：

#### 10.8.1 HTTP/2 Server Push（服务器主动推送）

**原理：**
- HTTP/2 允许服务器主动推送资源给客户端
- 服务器可以预测客户端需要的下一个 TS 片段，提前推送

**优势：**
- 减少往返次数（RTT）
- 可以提前推送，降低延迟
- 减少客户端请求次数

**劣势：**
- 浏览器支持有限（Chrome 已废弃）
- 可能推送不需要的资源，浪费带宽
- 实现复杂度较高

**适用场景：**
- 内网环境
- 对延迟要求极高的场景
- 不推荐用于生产环境（浏览器支持差）

#### 10.8.2 HTTP/3 (QUIC) 协议

**原理：**
- 基于 UDP 的传输协议
- 内置多路复用、0-RTT 连接建立
- 更好的拥塞控制和丢包恢复

**优势：**
- 连接建立更快（0-RTT）
- 更好的网络适应性
- 多路复用，减少队头阻塞

**劣势：**
- 需要服务器和客户端都支持 HTTP/3
- 部署复杂度高
- 部分网络环境可能不支持 UDP

**适用场景：**
- 高并发视频服务
- 移动网络环境
- 需要极致性能的场景

**实现建议：**
```go
// 需要支持 HTTP/3 的服务器（如 Caddy、Nginx 1.25+）
// 后端代理层可以继续使用 HTTP/1.1，由 Nginx 处理 HTTP/3
```

#### 10.8.3 WebSocket 流式传输

**原理：**
- 建立 WebSocket 长连接
- 服务器主动推送视频数据流
- 客户端实时接收并播放

**优势：**
- 实时性极强
- 可以精确控制传输节奏
- 支持双向通信

**劣势：**
- 需要维护长连接，服务器资源消耗大
- 实现复杂度高
- 不适合大规模并发
- 需要自定义协议

**适用场景：**
- 低延迟直播
- 小规模用户
- 需要精确控制的场景

**实现示例：**
```go
// WebSocket 流式传输示例（伪代码）
func handleVideoStream(conn *websocket.Conn, videoURL string) {
    resp, _ := http.Get(videoURL)
    defer resp.Body.Close()
    
    buffer := make([]byte, 8192)
    for {
        n, err := resp.Body.Read(buffer)
        if err != nil {
            break
        }
        conn.WriteMessage(websocket.BinaryMessage, buffer[:n])
    }
}
```

#### 10.8.4 CDN + 边缘缓存

**原理：**
- 使用 CDN 将视频内容缓存到边缘节点
- 用户从最近的节点获取视频
- 结合 Range 请求实现边下边播

**优势：**
- 大幅降低延迟（地理距离近）
- 减少源服务器压力
- 提升并发能力
- 支持 Range 请求

**劣势：**
- 需要 CDN 服务（成本）
- 缓存更新有延迟
- 配置复杂度较高

**适用场景：**
- 大规模视频服务
- 全球用户分布
- 对性能要求高的场景

**实现建议：**
```
用户 → CDN 边缘节点（支持 Range） → 源服务器
      ↓
   缓存 TS 片段
```

#### 10.8.5 智能预加载策略

**原理：**
- 分析用户观看行为
- 预测下一个需要的片段
- 提前请求并缓存

**优势：**
- 进一步降低延迟
- 提升播放流畅度
- 减少等待时间

**劣势：**
- 需要预测算法
- 可能浪费带宽（预测错误）
- 实现复杂度高

**实现示例：**
```javascript
// 前端智能预加载
class SmartPreloader {
  preloadNextSegments(currentIndex, playlist) {
    // 预加载当前片段后的 2-3 个片段
    for (let i = 1; i <= 3; i++) {
      const segment = playlist[currentIndex + i]
      if (segment) {
        this.preloadSegment(segment)
      }
    }
  }
}
```

#### 10.8.6 自适应码率 (ABR) + Range

**原理：**
- 根据网络状况动态切换视频质量
- 网络好时请求高清，网络差时请求标清
- 结合 Range 请求实现边下边播

**优势：**
- 自适应网络状况
- 提升用户体验
- 节省带宽（网络差时）

**劣势：**
- 需要多码率视频源
- 实现复杂度较高
- 需要播放器支持

**适用场景：**
- 移动网络环境
- 网络状况不稳定的场景
- 需要优化带宽使用的场景

**M3U8 示例：**
```m3u8
#EXTM3U
#EXT-X-STREAM-INF:BANDWIDTH=2000000,RESOLUTION=1280x720
playlist_720p.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=1000000,RESOLUTION=854x480
playlist_480p.m3u8
```

#### 10.8.7 P2P 传输（WebRTC）

**原理：**
- 使用 WebRTC 实现点对点传输
- 用户之间共享视频片段
- 减少服务器带宽压力

**优势：**
- 大幅减少服务器带宽
- 用户越多，速度越快
- 去中心化架构

**劣势：**
- 实现复杂度极高
- 需要 STUN/TURN 服务器
- 隐私和安全问题
- 不适合小规模场景

**适用场景：**
- 大规模视频平台
- 带宽成本敏感的场景
- 有技术团队支持

**代表产品：**
- Bittorrent Live
- WebTorrent

#### 10.8.8 方案对比总结

| 方案 | 延迟 | 复杂度 | 成本 | 适用场景 | 推荐度 |
|------|------|--------|------|----------|--------|
| **HTTP Range** | 低 | 低 | 低 | 通用场景 | ⭐⭐⭐⭐⭐ |
| **HTTP/2 Push** | 极低 | 中 | 低 | 内网/实验 | ⭐⭐ |
| **HTTP/3 (QUIC)** | 极低 | 高 | 中 | 高性能场景 | ⭐⭐⭐⭐ |
| **WebSocket** | 极低 | 高 | 中 | 小规模/直播 | ⭐⭐⭐ |
| **CDN + Range** | 极低 | 中 | 高 | 大规模服务 | ⭐⭐⭐⭐⭐ |
| **智能预加载** | 极低 | 中 | 低 | 优化体验 | ⭐⭐⭐⭐ |
| **ABR + Range** | 低 | 中 | 中 | 移动网络 | ⭐⭐⭐⭐ |
| **P2P (WebRTC)** | 低 | 极高 | 低 | 大规模平台 | ⭐⭐ |

#### 10.8.9 推荐方案组合

**当前阶段（小到中规模）：**
```
HTTP Range 请求 + 智能预加载
```
- 实现简单
- 性能优秀
- 成本低

**发展阶段（中到大规模）：**
```
CDN + HTTP Range + 智能预加载 + ABR
```
- CDN 降低延迟
- Range 实现边下边播
- 预加载提升体验
- ABR 适应网络

**成熟阶段（大规模）：**
```
CDN + HTTP/3 + Range + 智能预加载 + ABR + P2P（可选）
```
- 全链路优化
- 极致性能
- 成本优化

### 10.9 总结

Range 请求是实现视频边下边播的核心技术：
- **原理**：客户端请求文件的部分字节，服务器返回对应范围的数据
- **优势**：低延迟、低内存、支持拖拽、节省带宽
- **实现**：后端透传 Range 请求头，前端播放器自动利用
- **效果**：实现类似云播放器的流畅播放体验

**进阶方案建议：**
1. **短期**：继续使用 HTTP Range，添加智能预加载
2. **中期**：引入 CDN，结合 Range + 预加载 + ABR
3. **长期**：考虑 HTTP/3，根据业务规模决定是否引入 P2P

### 11.7 Nginx CORS 配置（跨域问题解决方案）

#### 11.7.1 问题描述

本地前端（`http://localhost:5173`）访问线上后端（`https://oldbai.top/gateway/api/v1/m3u8/proxy`）时，出现跨域错误：
```
Access to video at 'https://oldbai.top/gateway/api/v1/m3u8/proxy?url=...' 
from origin 'http://localhost:5173' has been blocked by CORS policy: 
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

#### 11.7.2 问题原因

1. **中间件在 Handler 之前执行**：当 `ApiEnabledMiddleware` 返回错误时，Handler 不会执行，导致后端设置的 CORS 头不会生效
2. **Nginx 代理层**：nginx 作为反向代理，需要确保后端设置的 CORS 头能正确传递，并在后端出错时提供兜底

#### 11.7.3 解决方案

**后端代码层面（已完成）：**
- ✅ `ApiEnabledMiddleware`：在返回错误时，对需要 CORS 的接口（`/m3u8/proxy`、`/videos/collect`）设置 CORS 头
- ✅ `M3u8ProxyHandler`：在 Handler 最开始设置 CORS 头
- ✅ `VideoCollectHandler`：添加 CORS 响应头

**Nginx 配置层面（需要部署）：**

在 `/gateway/` location 中添加 CORS 支持，作为兜底方案：

```nginx
location ^~ /gateway/ {
    proxy_pass http://127.0.0.1:20000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For "$proxy_add_x_forwarded_for";
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Forwarded-Host $host;
    proxy_set_header X-Forwarded-Prefix /gateway;

    # 添加proxy优化
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering on;
    proxy_buffer_size 4k;
    proxy_buffers 8 4k;

    # 对于需要 CORS 的接口，在 nginx 层面添加 CORS 头作为兜底
    # 使用 always 确保即使后端返回错误（如 400/403），也有 CORS 头
    set $cors_add 0;
    if ($request_uri ~* "^/gateway/api/v1/m3u8/proxy") {
        set $cors_add 1;
    }
    if ($request_uri ~* "^/gateway/api/v1/videos/collect") {
        set $cors_add 1;
    }
    if ($cors_add = 1) {
        add_header Access-Control-Allow-Origin "*" always;
        add_header Access-Control-Allow-Methods "GET, POST, OPTIONS" always;
        add_header Access-Control-Allow-Headers "Content-Type,Authorization,Range,X-Requested-With" always;
        add_header Access-Control-Max-Age "86400" always;
    }
    # m3u8 接口需要额外的 Expose-Headers
    if ($request_uri ~* "^/gateway/api/v1/m3u8/proxy") {
        add_header Access-Control-Expose-Headers "Content-Range,Accept-Ranges,Content-Length" always;
    }

    # 处理 OPTIONS 预检请求（针对需要 CORS 的接口）
    if ($request_method = OPTIONS) {
        set $cors_options 0;
        if ($request_uri ~* "^/gateway/api/v1/m3u8/proxy") {
            set $cors_options 1;
        }
        if ($request_uri ~* "^/gateway/api/v1/videos/collect") {
            set $cors_options 1;
        }
        if ($cors_options = 1) {
            add_header Access-Control-Allow-Origin "*" always;
            add_header Access-Control-Allow-Methods "GET, POST, OPTIONS" always;
            add_header Access-Control-Allow-Headers "Content-Type,Authorization,Range,X-Requested-With" always;
            add_header Access-Control-Max-Age "86400" always;
            add_header Content-Length 0;
            add_header Content-Type text/plain;
            return 204;
        }
    }

    # 添加proxy优化
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering on;
    proxy_buffer_size 4k;
    proxy_buffers 8 4k;
}
```

#### 11.7.4 配置说明

**关键配置项：**

1. **`proxy_pass_header`**：确保后端设置的 CORS 响应头能传递到客户端
2. **`add_header ... always`**：使用 `always` 参数，确保即使后端返回错误状态码（如 400/403），也能添加 CORS 头
3. **`X-Forwarded-*` 头**：确保后端能正确获取请求的协议、主机和前缀

**为什么需要 nginx 层面的 CORS？**

1. **中间件错误响应**：当 `ApiEnabledMiddleware` 返回错误时，Handler 不会执行，后端设置的 CORS 头可能不会生效
2. **兜底保障**：nginx 层面的 CORS 头作为兜底，确保即使后端出错，也能正确响应跨域请求
3. **OPTIONS 预检**：nginx 可以直接处理 OPTIONS 请求，减少后端压力

#### 11.7.5 部署步骤

1. 将更新后的配置复制到服务器的 nginx 配置文件（`/etc/nginx/sites-enabled/` 或相应位置）
2. 测试配置语法：`nginx -t`
3. 重载配置：`nginx -s reload` 或 `systemctl reload nginx`

#### 11.7.6 验证方法

部署后，可以通过以下方式验证：

```bash
# 测试 OPTIONS 预检请求
curl -X OPTIONS \
  -H "Origin: http://localhost:5173" \
  -H "Access-Control-Request-Method: GET" \
  -H "Access-Control-Request-Headers: Content-Type,Authorization,Range" \
  -v https://oldbai.top/gateway/api/v1/m3u8/proxy?url=test

# 应该看到响应头包含：
# Access-Control-Allow-Origin: *
# Access-Control-Allow-Methods: GET, POST, OPTIONS
# Access-Control-Allow-Headers: Content-Type,Authorization,Range,X-Requested-With
```

#### 11.7.7 注意事项

1. **nginx if 的限制**：nginx 的 `if` 语句有副作用，多个 `if` 语句可能不会按预期工作，建议使用 `map` 指令（需要在 http 块中定义）
2. **always 参数**：必须使用 `always` 参数，否则错误响应（4xx/5xx）不会添加 CORS 头
3. **路径匹配**：确保正则表达式正确匹配需要 CORS 的接口路径
4. **配置位置**：如果使用 `map` 指令，需要在主 `nginx.conf` 的 `http` 块中定义，不能在 server 块中

#### 11.7.8 完整配置示例（使用 map 指令）

如果使用 `map` 指令（更优雅），需要在主 `nginx.conf` 的 `http` 块中添加：

```nginx
http {
    # ... 其他配置 ...

    # 定义需要 CORS 的路径映射
    map $request_uri $cors_enabled {
        default 0;
        ~^/gateway/api/v1/m3u8/proxy 1;
        ~^/gateway/api/v1/videos/collect 1;
    }

    # ... server 块 ...
}
```

然后在 location 中使用 `$cors_enabled` 变量，代码更简洁。

### 12.2 数据库与数据模型设计

> 实际表结构以增量 SQL 为准，以下为字段规划草案；统一包含 `created_at/updated_at/deleted_at` BIGINT 秒级字段，软删除遵循全局规范。

- **表一：`blog_tag`（文章标签表）**
  - `id`：BIGINT，自增主键。
  - `name`：VARCHAR，标签名称（长度上限通过字典配置，逻辑层按字典校验，支持中文长度统计）。
  - `status`：TINYINT，启用状态（1=启用，0=禁用，枚举从 1 开始，走字典或系统常量）。
  - `remark`：VARCHAR，可选备注。
  - `created_at / updated_at / deleted_at`：BIGINT。

- **表二：`blog_article`（文章主表）**
  - `id`：BIGINT，自增主键。
  - `title`：VARCHAR，文章标题（最大长度通过字典控制）。
  - `content`：LONGTEXT，Markdown 原文内容（仅存储原始文本，不做富文本转换）。
  - `status`：TINYINT，文章业务状态（走字典，例如：1=草稿，2=待审核，3=审核通过-未上架，4=上架，5=下架）。
  - `audit_status`：TINYINT，审核状态（走字典，例如：1=未提交，2=待审核，3=审核通过，4=审核驳回）。
  - `cover`：VARCHAR，封面图片 URL，可为空（为空时前端/VO 逻辑按标题首字生成占位封面）。
  - `author_id`：BIGINT，作者后台用户 ID（取当前登录管理员 ID）。
  - `author_name`：VARCHAR，作者姓名快照（冗余，便于展示与审计）。
  - `publish_time`：BIGINT，文章上架时间（秒级时间戳，仅当状态为「上架」时非 0）。
  - `summary`：VARCHAR，可选摘要/简介字段（可选；公共列表 VO 按字典配置长度截断生成）。
  - `created_at / updated_at / deleted_at`：BIGINT。

- **表三：`blog_article_tag`（文章-标签关联表，多对多）**
  - `id`：BIGINT，自增主键。
  - `article_id`：BIGINT，文章 ID。
  - `tag_id`：BIGINT，标签 ID。
  - `created_at / updated_at / deleted_at`：BIGINT（允许软删除，避免误删历史关联）。

- **表四：`blog_article_audit`（文章审核记录表）**
  - `id`：BIGINT，自增主键。
  - `article_id`：BIGINT，文章 ID。
  - `audit_status`：TINYINT，本次审核结果（与文章审核状态共用字典）。
  - `audit_remark`：VARCHAR，审核意见。
  - `auditor_id`：BIGINT，审核人后台用户 ID。
  - `auditor_name`：VARCHAR，审核人姓名快照。
  - `created_at / updated_at / deleted_at`：BIGINT（`updated_at` 可与 `created_at` 一致）。

> 后续生成 Model 时，统一通过 goctl model + 自定义模板，接入 sqlx + cache；Repository 层聚合访问，避免在 Logic 中直接操作 Model。

### 12.3 字典与枚举设计（要求 1/2/3/4/7）

> 所有业务可变枚举、长度配置、展示配置统一走数据字典；**枚举值统一从 1 开始**，禁止使用 0；系统级常量仍放在 `internal/consts`。

- **字典类型一：`blog_article_status`（文章业务状态）**
  - value 从 1 开始，例如：
    - 1：草稿
    - 2：待审核
    - 3：审核通过-未上架
    - 4：上架
    - 5：下架
  - 用途：后台文章列表筛选、状态展示；逻辑层校验允许的状态流转（禁止越级跳转）。

- **字典类型二：`blog_article_audit_status`（文章审核状态）**
  - 1：未提交
  - 2：待审核
  - 3：审核通过
  - 4：审核驳回
  - 用途：文章提交审核、审核操作、公共列表过滤（仅展示审核通过 + 上架）。

- **字典类型三：`blog_tag_name_max_length`（标签名称最大长度）**
  - 例如：`value=10` 表示最多 10 个「字符单位」（在后端按 Unicode rune 统计，兼容中文）。
  - 逻辑：标签创建/编辑时，后端从字典读取最大长度，校验 `name` 长度；前端可通过字典做输入限制。

- **字典类型四：`blog_article_title_max_length`（文章标题最大长度）**
  - `value` 为正整数，表示标题最大长度（同样按 rune 统计长度）。
  - 逻辑：文章创建/编辑接口在 Logic 层校验标题长度，超长返回业务错误码。

- **字典类型五：`blog_article_summary_length`（公共展示内容截断长度）**
  - `value` 为正整数，例如 120/200 等，表示公共列表/详情摘要展示的最大字符数。
  - 用途：公共文章列表/详情 VO 组装时，根据该长度对 Markdown 渲染后的纯文本或原文进行截断。

- **（可选）字典类型六：`blog_article_cover_strategy`（封面策略）**
  - 1：无封面使用标题首字
  - 2：无封面使用系统默认图片
  - 后续如有需要可扩展，目前实现上约定使用「标题首字」，无需前端额外配置。 

- **审计类型扩展：在现有 `audit_log_type`（或同类审计类型字典）中新增博客相关字典项**
  - 新增字典项示例（value 从当前最大值递增，且 >0）：
    - `blog_article_audit`：文章审核
    - `blog_article_unpublish`：文章下架
  - 用途：文章审核与下架操作写入审计日志时，使用上述审计类型，便于在审计日志中区分来源模块与操作类型。

> 上述字典类型及字典项将通过增量 SQL 写入 `admin_dict_type/admin_dict_item`，ID 从现有最大值递增，遵守 1 开始与不冲突原则。

### 12.4 后端接口与分层设计

> 所有接口按现有规范在 `admin-server/api/admin.api` 中定义，使用 goctl 生成 Handler/Logic/Types，Repository 手工聚合；权限、菜单、接口映射统一通过增量 SQL 初始化。

- **标签管理接口（后台，需登录 + 权限）**
  - 分组建议：`blog_tag`。
  - 中间件：`AuthMiddleware, PermissionMiddleware, OperationLogMiddleware`。
  - API（草案，具体路径在 `.api` 中统一）：
    - `GET /api/v1/blog/tags`：标签列表（分页 + 按名称模糊搜索 + 状态筛选）。
    - `GET /api/v1/blog/tags/options`：标签下拉选项（仅返回启用标签，用于文章编辑页标签多选；支持 `limit` 可选参数）。
    - `POST /api/v1/blog/tags`：新增标签（校验名称长度、唯一性）。
    - `PUT /api/v1/blog/tags`：编辑标签（含启用/禁用）。
    - `DELETE /api/v1/blog/tags`：删除标签（软删，若有文章关联可限制为逻辑禁用而非删除）。

- **文章管理接口（后台，需登录 + 权限）**
  - 分组建议：`blog_article`。
  - 中间件：`AuthMiddleware, PermissionMiddleware, OperationLogMiddleware`。
  - 核心接口（示例）：
    - `GET /api/v1/blog/articles`：文章列表（分页 + 状态/审核状态/标签筛选 + 关键词搜索）。
    - `GET /api/v1/blog/articles/:id`：文章详情（后台查看，包含完整 Markdown 内容与审核记录）。
    - `POST /api/v1/blog/articles`：创建文章（初始状态为「草稿」，自动设置作者信息）。
    - `PUT /api/v1/blog/articles`：编辑文章（仅草稿或审核驳回状态允许编辑正文；已审核通过建议只允许部分字段编辑）。
    - `POST /api/v1/blog/articles/submit`：提交审核（草稿/驳回 → 待审核）。
    - `POST /api/v1/blog/articles/publish`：上架文章（仅审核通过的文章可上架，写入 `publish_time`）。
    - `POST /api/v1/blog/articles/unpublish`：下架文章（状态改为「下架」，公共列表不再展示）。

- **文章审核接口（后台，需登录 + 权限）**
  - 分组建议：`blog_article_audit`（可与文章管理同组，也可独立分组）。
  - 中间件：`AuthMiddleware, PermissionMiddleware, OperationLogMiddleware, AuditMiddleware(复用审计工具)`（审计可选，通过业务逻辑手动调用 `pkg/audit`）。
  - 核心接口：
    - `GET /api/v1/blog/articles/audits`：审核记录列表（支持按文章、审核人、时间范围筛选）。
    - `POST /api/v1/blog/articles/audit`：执行审核操作（参数包括：`articleId`、`result`=通过/驳回、`remark`）。
      - 逻辑：
        - 校验文章当前状态为「待审核」。
        - 记录审核记录到 `blog_article_audit`。
        - 更新 `blog_article.audit_status` 与 `status`（通过时设置为「审核通过-未上架」，驳回则为「审核驳回」）。
        - 记录审计日志（使用 `pkg/audit`，审计类型为「文章审核」）。
    - `POST /api/v1/blog/articles/audit/unpublish`：审核员执行下架操作（与普通上架/下架区分权限与审计类型）。
      - 逻辑：
        - 校验文章当前状态为「上架」。
        - 将文章状态更新为「下架」，可记录一条审核记录（类型为「下架」），方便追踪是谁下架了文章。
        - 写入审计日志（审计类型为「文章下架」，包含原状态、目标状态与原因备注）。

- **公共文章接口（无需登录 / 公共访问）**
  - 分组建议：`public_blog`，与公共视频管理类似。
  - 中间件：`ApiEnabledMiddleware, PublicOperationLogMiddleware`（如有需要可增加简单防刷策略，与现有公共接口保持一致）。
  - 核心接口：
    - `GET /api/v1/public/blog/articles`：公共文章列表。
      - 过滤条件：仅 `status=上架` 且 `audit_status=审核通过`；支持按标签、关键字、时间排序（如按发布时间倒序）。
      - 返回 VO：不返回完整 Markdown，只返回截断后的摘要（按字典长度）、封面、标题、作者名、发布时间、标签列表、文章 ID 等。
    - `GET /api/v1/public/blog/articles/:id`：公共文章详情。
      - 校验：同样只允许已审核 + 上架的文章访问（即使传入 ID 存在但未上架，也返回业务错误）。
      - 返回 VO：
        - 标题、作者、发布时间、标签列表、封面 URL（若无封面则按策略生成占位字段供前端渲染）、完整 Markdown 内容。
        - 可额外返回「推荐阅读」/上一篇下一篇 ID（后续扩展）。

- **通用打点上报接口（PV/UV/VV/IP，文章与视频共用）**
  - 分组建议：`metric` 或 `stat`（与业务解耦，作为通用埋点模块）。
  - 中间件：根据场景区分：
    - 公共打点（文章/视频公开访问）：`ApiEnabledMiddleware, PublicOperationLogMiddleware`。
    - 管理后台访问打点（如有需要）：`AuthMiddleware, ApiEnabledMiddleware, OperationLogMiddleware`。
  - 接口设计（示例，后续在 `.api` 中细化）：
    - `POST /api/v1/metrics/report`：通用 PV/UV/VV/IP 上报接口。
      - 请求体建议字段：
        - `module`：业务模块标识（如：`blog_article_list`、`blog_article_detail`、`video_list`、`video_detail` 等）。
        - `bizId`：业务主键 ID（如文章 ID 或视频 ID，列表页可为空或约定特殊值）。
        - `event`：事件类型（如 `view` / `play` 等，首期可统一为 `view`）。
        - `extra`：可选扩展字段（JSON 字符串或 map，用于记录来源、终端类型等）。
      - 统计口径说明：
        - **PV（Page View）**：同一用户/同一 IP 多次访问均计入，按记录条数统计。
        - **UV（Unique Visitor）**：按访客唯一键（如登录用户 ID 或匿名访客 cookie）去重后统计。
        - **VV（Visit/Video View）**：可按会话维度或业务约定统计（例如一次打开详情页视为一次 VV，与 PV 略有差异，具体规则在实现阶段细化）。
        - **IP**：按客户端 IP 维度统计访问量与独立 IP 数。
      - 数据落库方案（首期建议）：
        - 新增通用埋点表（例如 `metric_event`），记录原始事件；在后续异步任务或离线统计中聚合出 PV/UV/VV/IP 统计结果。
        - 为避免主流程阻塞，可在 Logic 层异步写入（结合现有异步任务或简单 goroutine + channel），具体实现待后续「埋点/统计」模块方案细化。

### 12.5 关键业务规则与流程说明

- **状态流转规则（结合需求 2）**
  - 初始创建：`status=草稿`，`audit_status=未提交`。
  - 提交审核：仅当 `status=草稿` 或 `audit_status=审核驳回` 时允许 → `audit_status=待审核`，`status` 可保持为草稿或统一为待审核。
  - 审核通过：`audit_status=审核通过`，`status` 更新为「审核通过-未上架」。
  - 上架：仅当 `audit_status=审核通过` 时允许：`status=上架`，写入 `publish_time`。
  - 下架：仅当 `status=上架` 时允许 → `status=下架`，`publish_time` 可保留历史值或置 0（建议保留以支持历史统计）。

- **文章编辑流程规则（新增）**
  - **编辑权限控制：**
    - 上架状态（`status=4`）：不可编辑，后端返回 `403 Forbidden` 错误，前端显示只读模式。
    - 其他状态（草稿、待审核、审核通过-未上架、下架）：允许编辑。
  - **编辑后状态流转：**
    - 草稿（`status=1`）→ 编辑保存后：保持草稿（`status=1`），审核状态保持未提交（`audit_status=1`）。
    - 待审核（`status=2`）→ 编辑保存后：转为待审核（`status=2`），审核状态重置为待审核（`audit_status=2`）。
    - 审核通过-未上架（`status=3`）→ 编辑保存后：转为待审核（`status=2`），审核状态重置为待审核（`audit_status=2`）。
    - 下架（`status=5`）→ 编辑保存后：转为待审核（`status=2`），审核状态重置为待审核（`audit_status=2`）。
  - **实现位置：**
    - 后端：`admin-server/internal/logic/blog_article/blogarticleupdatelogic.go`，在 `BlogArticleUpdate` 方法中实现状态检查和自动流转。
    - 前端：`admin-frontend/src/views/blog/BlogArticleEdit.vue`，根据文章状态控制表单可编辑性和按钮显示。

- **标签约束（结合需求 3）**
  - 后端在创建/编辑文章时强制校验：必须至少关联 1 个启用状态的标签。
  - 标签名称长度校验：从 `blog_tag_name_max_length` 字典读取上限，以 rune 计数，防止中文长度问题。

- **标题与内容长度约束（结合需求 4/7）**
  - 标题长度：从 `blog_article_title_max_length` 字典读取上限，在 Logic 层校验。
  - 列表摘要长度：从 `blog_article_summary_length` 读取，公共文章列表/详情中对内容进行截断：
    - 从 Markdown 文本中提取纯文本（可在后端简单做去 Markdown 标记，或交由前端处理，方案评审时决定）。
    - 截断并返回至 VO 字段 `summary`，供前端展示。

- **封面策略（结合需求 8）**
  - 若 `cover` 字段非空：直接返回。
  - 若 `cover` 为空：公共 VO 中返回一个 `coverFallback` 字段，内容为标题首个字符（按 rune 截取），方便前端生成首字占位封面。
  - 后续如引入系统默认封面，可在字典或配置中扩展。 

- **图片上传与文件系统集成（结合需求 9）**
  - 文章编辑页面中的图片上传统一走现有文件模块接口（例如 `/api/v1/files/upload` 或 SDK 上传接口），复用现有文件记录与权限控制。
  - 文章内容中仅存储图片 URL（或文件 ID 转换后的访问 URL），不单独为博客模块设计文件表。
  - 如有需要，可在文章详情 VO 中追加已引用图片列表（通过简单解析或约定格式实现，非首期必需）。

- **作者信息（结合需求 6）**
  - 创建/编辑文章时，从当前登录用户上下文获取 `userId/userName`，写入 `author_id/author_name`。
  - 后续如支持多人协作，可扩展「最后编辑人」字段或编辑记录表，当前阶段不实现。

- **审核与审计日志联动（结合补充需求 1/2/3）**
  - 所有文章审核相关操作（通过/驳回）必须记录到审计日志：
    - 审计类型：使用新扩展的字典项（例如 `blog_article_audit`）。
    - 审计内容：包含文章 ID、标题快照、原审核状态、目标审核状态、审核意见、审核员 ID/姓名等。
  - 审核员具备对文章进行审核与下架的权限：
    - 在权限与菜单初始化 SQL 中，为「文章审核」与「文章下架（审核下架）」分别配置独立的权限编码，便于精细授权。
    - 当审核员执行下架操作时，同样记录审计日志（审计类型例如 `blog_article_unpublish`），审计内容中记录下架原因。

### 12.6 前后端交互与 Markdown 特殊字符处理

- **Markdown 内容传输**
  - 前后端通过 JSON Body 传输 `content` 字段，类型为字符串，**不做任何 HTML 转义或富文本转换**，由前端在展示时使用 Markdown 渲染组件。
  - 注意：
    - 后端在存储前禁止对 Markdown 内容进行 `html/template` 转义，以免破坏原始文本；仅在输出到 HTML 时由前端负责安全过滤。
    - API 文档说明中明确：`content` 支持 `#`、`*`、`````、`<`、`>`、`&` 等特殊字符，后端需确保 JSON 编解码正确处理。

- **安全与长度控制**
  - 后端对 `title`、`summary`、`remark` 等短文本字段执行长度校验与基础字符校验（如禁止极端长度或明显恶意脚本片段）。
  - Markdown 内容本身作为长文本，只做总长度限制（可通过配置或字典扩展，当前阶段可先不限或设置较大上限）。

- **前端渲染约定（概要）**
  - 管理后台文章编辑使用 Markdown 编辑器（支持图片上传 Hook，调用现有文件上传接口），保存时直接提交 Markdown 文本。
  - 公共页面使用安全的 Markdown 渲染库，默认禁止内联脚本执行，链接打开方式与白名单策略待前端文档细化。

### 13.2 数据库设计

#### 13.2.1 友情链接表（`blog_friend_link`）

```sql
CREATE TABLE IF NOT EXISTS `blog_friend_link` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键 ID',
  `name` VARCHAR(64) NOT NULL COMMENT '链接名称（最大15个中文字符，通过字典配置）',
  `url` VARCHAR(255) NOT NULL COMMENT '目标链接（最大255字符，通过字典配置）',
  `remark` VARCHAR(512) DEFAULT '' COMMENT '备注（最大127个中文字符，通过字典配置）',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1=启用，2=禁用（字典：blog_friend_link_status）',
  `order_num` INT NOT NULL DEFAULT 0 COMMENT '排序值（数字越小越靠前）',
  `created_at` BIGINT NOT NULL DEFAULT 0 COMMENT '创建时间(秒级时间戳)',
  `updated_at` BIGINT NOT NULL DEFAULT 0 COMMENT '更新时间(秒级时间戳)',
  `deleted_at` BIGINT NOT NULL DEFAULT 0 COMMENT '删除时间(秒级时间戳,0表示未删除)',
  PRIMARY KEY (`id`),
  KEY `idx_blog_friend_link_status` (`status`),
  KEY `idx_blog_friend_link_order_num` (`order_num`),
```

## 14. 博客页面改造所需后端接口补充

### 14.1 需求概述

根据前端博客页面改造方案（参考 `docs/前端开发进度.md` 第11章），需要补充以下后端接口：

1. **获取超级管理员个人信息接口**：用于博客页面右侧侧边栏显示作者信息
2. **获取文章统计接口**：用于显示文章总数等统计信息
3. **获取相邻文章接口**：用于文章详情页的上一页/下一页导航

### 14.2 接口设计

#### 14.2.1 获取超级管理员个人信息接口

**接口路径：** `GET /api/v1/public/blog/author-info`

**接口说明：**
- 获取超级管理员（`admin_user.id = 1`）的个人信息
- 仅返回公开信息：昵称、头像、个性签名
- 不返回敏感信息：用户名、密码等

**请求参数：** 无

**响应结构：**
```go
type PublicBlogAuthorInfoResp struct {
    Id        uint64 `json:"id"`         // 用户ID（固定为1）
    Nickname  string `json:"nickname"`   // 昵称
    Avatar    string `json:"avatar"`     // 头像URL
    Signature string `json:"signature"` // 个性签名
}
```

**实现要点：**
- 从 `admin_user` 表查询 `id = 1` 的记录
- 仅返回公开字段，过滤敏感信息
- 如果用户不存在或已删除，返回默认值或错误

#### 14.2.2 获取文章统计接口

**接口路径：** `GET /api/v1/public/blog/article-stats`

**接口说明：**
- 获取博客文章的统计信息
- 统计已发布（审核通过 + 上架）的文章总数

**请求参数：** 无

**响应结构：**
```go
type PublicBlogArticleStatsResp struct {
    TotalArticles int64 `json:"totalArticles"` // 文章总数（已发布）
}
```

**实现要点：**
- 从 `blog_article` 表统计：`status = 4`（已发布）且 `audit_status = 2`（审核通过）且 `deleted_at = 0` 的记录数
- 使用 COUNT 查询，性能优化

#### 14.2.3 获取相邻文章接口

**接口路径：** 
- `GET /api/v1/public/blog/articles/prev`：获取上一篇文章
- `GET /api/v1/public/blog/articles/next`：获取下一篇文章

**接口说明：**
- 根据当前文章ID，获取相邻的已发布文章
- 排序规则：按发布时间倒序（`publish_time DESC`）
- 上一篇文章：发布时间早于当前文章
- 下一篇文章：发布时间晚于当前文章

**请求参数：**
```go
type PublicBlogArticlePrevReq struct {
    Id uint64 `json:"id,optional" form:"id,optional"` // 当前文章ID
}

type PublicBlogArticleNextReq struct {
    Id uint64 `json:"id,optional" form:"id,optional"` // 当前文章ID
}
```

**响应结构：**
```go
type PublicBlogArticlePrevResp struct {
    Id          uint64   `json:"id"`          // 文章ID
    Title       string   `json:"title"`       // 标题
    PublishTime int64    `json:"publishTime"` // 发布时间
}

type PublicBlogArticleNextResp struct {
    Id          uint64   `json:"id"`          // 文章ID
    Title       string   `json:"title"`       // 标题
    PublishTime int64    `json:"publishTime"` // 发布时间
}
```

**实现要点：**
- 查询条件：`status = 4`（已发布）且 `audit_status = 2`（审核通过）且 `deleted_at = 0`
- 上一篇文章：`publish_time < 当前文章publish_time`，按 `publish_time DESC` 排序，取第一条
- 下一篇文章：`publish_time > 当前文章publish_time`，按 `publish_time ASC` 排序，取第一条
- 如果不存在相邻文章，返回空值（前端判断后隐藏导航按钮）
