# 10 · 开发期执行策略：直接执行 / 事后 review / 真正停下

## 这篇文档的性质

**这是用户对本轮重构的明确澄清，覆盖（override）`AGENTS.md` 第 6 节"何时必须停下来问用户"的默认口径，但只在本轮重构的开发阶段生效，且只对本文档列出的具体事项生效。**

`AGENTS.md` 第 6 节默认要求：任何 `generate-*.sh` 脚本的执行、任何数据库 SQL 的实际执行都必须由用户亲自动手，AI 不得代替执行。这条默认规则是为日常新增业务模块场景设计的（每次生成都会产生需要人工核对的骨架代码/挂到正式菜单分类的数据）。但本轮重构体量大（161 个 logic 文件、9 个域、约 14 周），如果每次内部重构性质的 SQL/生成脚本执行都要停下来等确认，会把一次可以基本自动化推进的大规模机械重构拖成人工瓶颈——用户已明确要求这一轮**开发阶段**放开这条限制。

**不要做的事**：不要因为读了这篇文档就认为 `AGENTS.md` 第 6 节已经被永久修改了。第 6 节原文本身**不在本轮改动范围内**，是否要把这里的口径正式写回 `AGENTS.md`/`.cursor/rules`，是 Phase 3 文档收尾时要向用户提出的一个决定点（见 `13-rules-sync-checklist.md` Phase-3-end 批次），不能自己在 Phase 1/2 执行过程中顺手改了 `AGENTS.md` 第 6 节——那属于擅自修改全局规则，不属于"开发期直接执行"的授权范围。

## 1. 开发期可以直接执行，事后 review，不阻塞

以下事项 AI 可以在开发过程中直接执行，不需要先停下来问用户确认，执行完之后作为正常 diff 交给用户日常 review（即：这些操作本身不是"必须停下的检查点"，但操作产生的代码/数据变更仍然会被用户看到并可以事后指出问题）：

1. **`generate-*.sh` 系列脚本的实际执行**：`generate-sql.sh`、`generate-model.sh`、`generate-api.sh`、`generate-ts.sh`，以及 Phase 2 新增的 `generate-rpc.sh`、Phase 3 新增的 `generate-swagger.sh`（见 `12-scripts-standardization.md`）。这些脚本本身会生成/覆盖文件，`AGENTS.md` 默认要求用户亲自运行——本轮开发阶段这条放开，AI 可以直接跑。
2. **本地/开发环境数据库的真实 SQL 执行**：建表、字典初始化、权限初始化、RBAC 授权数据，以及 Phase 2 的库拆分迁移 SQL——前提是目标库是本地/开发环境（连接串指向本机或团队约定的 dev 库，不是生产 `/etc/work/mysql.json` 指向的库）。
3. **`make wire` 以及 `goctl rpc` 生成产物的提交**：生成后的 diff 留给用户日常 review，不需要生成前先问"要不要生成"。
4. **本地 `docker compose up` 验证**：如果执行环境里有 Docker，AI 可以自己跑 `docker compose up` 验证 09 篇设计的 compose 文件确实能拉起服务；如果环境里没有 Docker（很可能是当前场景），如实告知用户"这一步需要你本地验证一下"，不要假装验证过。
5. **`golangci-lint` 存量问题的自动修复**：能安全修的（`gofmt`/`goimports` 类、明确的 `ineffassign`/`unused` 未使用变量）直接修；判断不了是否安全（可能改变行为的）记录到 `11-descoped.md` 或对应 Phase 文档里，汇总一次性给用户 review，不要为了"清零 lint 问题"强行改动可能有风险的代码。
6. **本地开发用的 JWT 密钥 / 服务间临时凭证**：可以自动生成随机值写进本地 `.env`/`docker-compose.yml` 默认值，但**必须**在代码/配置注释里写清楚"仅供本地开发，生产环境必须通过环境变量覆盖，不能沿用此值"（`09-ci-cd-and-deployability.md` 第 2 节的 compose 示例已经这样做了）。

## 2. 必须真正停下来问用户的情况

以下事项是产品/业务判断，不是技术实现细节，AI 判断不了、也不应该替用户拍板：

1. **Chat onboarding 异步化的具体触发时机等产品/体验取舍**——04-domain-iam-chat.md 任务 1 已经确认"异步、尽力而为"的技术方向，但比如"新用户注册后多久内看到默认群消息算合理"这类体验细节，实现前仍需要跟用户确认一次。已有初步方向不等于可以直接实现所有细节。
2. **任何触及 `/etc/work/mysql.json`、`/etc/work/redis.json` 真实内容，或生产环境部署动作本身**——这条和 `AGENTS.md` 第 6 节完全一致，本轮**没有**放开，因为这直接影响生产环境真实数据/服务可用性，风险性质和"本地开发库 SQL"完全不同。
3. **生产环境真实密钥/凭证取值**——AI 只能在 `14-production-deployment-checklist.md` 里写清楚"你需要生成并设置什么值、放在哪里"，不能替用户生成生产密钥（哪怕只是"先生成一个占位值，用户之后再换"也不行——生产密钥一旦生成过就有被记录/泄露的风险，必须由用户自己在部署环境里生成）。
4. **Phase 2 每次服务拆分后，是否真的按 compose 切到生产**——涉及真实停机窗口，需要用户拍板时机，不能因为"代码写完了、测试也过了"就自行判断"可以上生产了"。
5. **任何 AI 自己判断不准的架构/命名/取舍点**——这是通用兜底原则，**拿不准就问，不是逢清单必停**。本文档列出的 1-4 条是已知必停的类别，但不是穷举；如果推进过程中遇到本文档没预见到的、明显涉及产品决策或生产风险的选择，同样要停下来问，不要因为"没在清单里"就自行决定。

## 3. 这条策略覆盖的范围边界

- 只适用于 `admin-server` 本轮重构（Phase 1-3）期间的开发操作。
- 不适用于本轮重构之外的日常新增业务模块开发——那仍然按 `AGENTS.md` 第 6 节的默认口径，`generate-*.sh` 照常需要用户亲自执行确认。
- 不因为这篇文档的存在而自动延伸到本轮重构之外的其他改动类型（比如不能因为这里放开了"本地 SQL 执行"就推广到"生产 SQL 也可以先执行再说"——第 2 节第 2 条已经明确排除生产环境）。

## 完成的定义

本篇是政策文档，没有代码层面的"完成"标准，但有一条持续生效的执行要求：**每次真正遇到第 2 节列出的停下来的情况，都要在向用户提问时说明"根据 10-dev-execution-and-review-points.md 第 2 节第 N 条，这里需要你确认"，让用户能对照文档核实这次停下来是不是合理的**，避免"该停的没停"或"不该停的乱停"两种偏差。Phase 3 文档收尾时，回头检查实际执行过程中是否出现过"本该停但没停"或"本该直接做但反复问"的情况，作为是否要把本文档内容正式写回 `AGENTS.md` 的参考依据之一。
