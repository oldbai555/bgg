**1. 现状核查确认文档前提成立**：`grep -rn "Telemetry|opentelemetry|otel"` 业务代码零命中，`go.mod` 已通过 go-zero 1.9.3 间接带全套 `otel*` 依赖，接入不需要 `go get` 任何新包。核实 go-zero 框架内部机制：`service.ServiceConf.SetUp()`（`rest.MustNewServer`/`zrpc.MustNewServer` 内部调用）本身就会 `logx.SetUp(sc.Log)` + `trace.StartAgent(sc.Telemetry)`，`logx.LogConf.Encoding` 默认值就是 `json`——五个 RPC 服务此前从未显式调用过 `logx.SetUp`，只有 gateway 的 `admin.go` 显式调了一次 `logx.SetUp(logx.LogConf{Encoding: "plain"})`（`logx.SetUp` 内部 `setupOnce.Do` 保证只有第一次调用生效），这是全仓库唯一还在用纯文本日志的入口，符合文档"现状核查"一节的描述。

**2. `pkg/logging` 新增（`setup.go` + `fields.go`）**：`Setup(serviceName string)` 封装 `logx.SetUp(logx.LogConf{ServiceName: serviceName, Encoding: "json", Level: "info"})`；`fields.go` 定义 `FieldTraceID`/`FieldSpanID`/`FieldService`/`FieldUserID` 四个标准字段常量（前三个由 go-zero 框架自动注入，不需要业务代码拼接）。

**3. 六处 `main` 函数机械替换**：`admin.go` 把内联的 `logx.SetUp(logx.LogConf{Encoding:"plain"})` 换成 `logging.Setup("gateway")`；`services/{iam,task,sdk,chat,content}/*.go` 五个此前完全没有日志初始化调用的 RPC 服务，各自在 `conf.MustLoad` 之后、`svc.NewServiceContext`/`zrpc.MustNewServer` 之前补一次 `logging.Setup("<service>-rpc")` 调用（`iam-rpc`/`task-rpc`/`sdk-rpc`/`chat-rpc`/`content-rpc`），不是"改现有调用"而是"补一次缺失的显式初始化"，让 `ServiceName` 字段可控，不依赖 `RpcServerConf.Name`（`iam.rpc` 这种带点号的内部命名）隐式带出。

**4. 六个 `etc/*.yaml` 加 `Telemetry` 配置块**：`Name` 分别为 `gateway`/`iam-rpc`/`task-rpc`/`sdk-rpc`/`chat-rpc`/`content-rpc`（与 `pkg/logging.Setup` 传入的 serviceName 保持一致，便于日志 `service` 字段和未来 trace 后端的服务名对上），`Sampler: 1.0`，`Batcher: otlpgrpc`。**发现并修正了文档 2.1 节示例本身的一处配置错误**：文档给的示例是 `Endpoint: http://127.0.0.1:4318/v1/traces` 配 `Batcher: otlpgrpc`——`otlptracegrpc.WithEndpoint` 期望纯 `host:port` 的 gRPC target 语法，带 scheme+path 的 URL 会在真实起服务时触发 `dial tcp: address ...: too many colons in address` 报错（冒烟验证时在 iam-rpc 真实进程日志里现场复现），而不是文档描述的"未部署 collector 时静默丢弃"那种优雅降级——这是一个配置语法错误，不是"没有监听方"的正常降级路径。改成 `Endpoint: 127.0.0.1:4317`（otlpgrpc 标准端口，纯 host:port）后，未部署 collector 时的实际报错退化成预期的 `connection refused`（nonblock，不影响启动），符合文档原意。六个 yaml 文件均已按修正后的写法落地。

**5. `pkg/audit.go` 补 `user_id` 结构化字段（文档 3.4 节"唯一需要新写代码的地方"）**：`RecordAuditLog` 的成功/失败日志行改用 `logx.WithContext(ctx).WithFields(logx.Field(logging.FieldUserID, userId))` 构造的 logger，而不是原来的包级 `logx.Errorf`/`logx.Infof`——这是全仓库审计日志的唯一入口，是文档所说"有真实排查价值的地方才加"的判断结果，不要求其余 130+ 个 Logic 文件跟进。

**6. 验证**：`go build ./...`/`go vet ./...`/`go test ./...`/`golangci-lint run ./...` 全仓库全绿，改动范围严格是"纯配置 + 一个新增小包 + 六处 main 函数机械替换 + 一处日志字段"，未触碰任何业务逻辑分支。

**7. 真实跨进程冒烟验证（借用户远程库 `oldbai` + 本机 Redis，和 Phase 2 五次拆分同一个库/同一个 mysql MCP 只读连接）**：编译 `gateway`/`iam-rpc` 两个临时二进制，起两个真实进程（`iam-rpc` 连 oldbai 真实 MySQL + 本机 Redis，`gateway` 连本机 Redis + 指向 iam-rpc 的 `IAM_RPC_ENDPOINT`/`IAM_CALLBACK_RPC_ENDPOINT`，其余四个 RPC endpoint 指向未占用端口不验证），请求 `GET /api/v1/ping`（触发 gateway → iam-rpc 的 `IamRPC.Ping`，内部真实查了一次 MySQL `SELECT 1` + Redis）。**验证结果**：gateway 和 iam-rpc 两边 stdout 全部是合法 JSON（`@timestamp`/`level`/`caller`/`content` 标准字段），且同一次请求在两边日志里能查到完全相同的 `trace` 字段值（如 `ea043aff8d52b77716552177549a21bd`），包括 iam-rpc 内部的 SQL 查询日志行也带着同一个 trace 值——跨进程 trace 关联链路验证通过，完全符合文档"完成的定义"第 4 条的验证标准。验证完成后终止两个测试进程，删除临时二进制和临时配置文件，未产生任何持久化测试数据（只读 ping，未触发任何写路径）。

**提交时本仓库的 pre-commit AI code review 钩子（Gentleman Guardian Angel）拦下并顺手修复了三类问题**（钩子对照 `AGENTS.md` 独立复核了本轮 diff，不是本轮主动排查发现的）：
- `services/iam/iam.go` 缺少 JWT/SDK-RPC/Chat-RPC 的 fail-fast 校验——iam-rpc 承接了 `Login`/`Refresh` 的 JWT 签发，且内部 `TaskCallbackServer`/消费者依赖 `SdkRPC`/`ChatRPC`，空配置下服务仍会"成功启动"、只在真正调用时才报错，和 `admin.go` 已经建立的 fail-fast 约定不一致。补齐 `JWT.AccessSecret/RefreshSecret`、`SdkRpc.Endpoints`、`ChatRpc.Endpoints` 三组启动期校验，写法与 `admin.go` 对齐。
- `pkg/audit.go` 的 `RecordAuditLog` 在异步 goroutine 里对 `IamCallbackRPC.RecordAuditLog` 用的是 `context.Background()`，本轮刚验证过的跨进程 trace 关联在这条异步路径上其实是断的——`context.Background()` 丢掉了 ctx 里 Telemetry 中间件放的 span context，iam-rpc 侧收到审计 RPC 时会开一条新 trace，没法用触发它的那次 HTTP 请求的 trace-id 关联。改用 `context.WithoutCancel(ctx)`（Go 1.24 标准库，本仓库 `go.mod` 已是 1.24）：既保留 ctx 里的 trace span context 用于传播，又摘掉请求生命周期的取消信号，避免 handler 已经返回、原始请求 ctx 被取消导致这次异步审计调用跟着夭折。
- `services/{task,chat,content}/etc/*.yaml` 三处关于 `TaskCallbackRpc`/`IamCallbackRpc` 的注释仍停留在 iam-rpc 拆分前的描述（"单体内嵌 TaskCallback/IamCallback server"、"docker-compose 用服务名 app:9001"、"iam 域还没拆分成独立服务前的临时方案"）——iam-rpc 拆分（见上一条会话记录）早已把这两个 server 搬进 iam-rpc 进程，`docker-compose.yml` 也已经把这两个 endpoint 改指向 `iam:8081`，过时注释会误导后续按注释手动配置 endpoint。三处均按当前架构改写。
