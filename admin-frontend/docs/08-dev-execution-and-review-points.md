# 开发期执行策略：直接执行 vs 事后 review vs 必须停下（Phase 1-3 全程适用）

> 参照 `admin-server/docs/10-dev-execution-and-review-points.md` 的先例定制到前端场景。本文档是 `00-workflow.md`"何时必须停下来问用户"一节针对 admin-frontend 本轮重构项目的**项目级临时澄清**，不是对根规则的静默修改；范围仅限本重构项目的开发阶段，不延伸到日常业务模块开发，也不延伸到本项目之外的其他前端改动。

## 1. 可以直接执行、事后随 diff 走查（不需要逐次停下确认）

用户已明确"时间足够，可以放心大胆重构，要改得彻底"，以下操作在本项目范围内直接执行：

1. `views/`/`api/` 目录重组、文件改名/移动（`git mv`）。
2. 新建 API wrapper 文件、修改调用点 import 路径。
3. `request.ts`、`router/index.ts` 的类型安全/去重改造（`01` 号文档范围）。
4. `composables/`/`hooks/` 合并。
5. `vitest` 引入、测试用例编写。
6. 死代码删除（`views/temp/*`）、死配置清理（`api:gen`、Prettier 残留注释）——前提是已按 `07` 号文档确认是真正的死代码/死配置，不是似是而非的猜测。
7. `websocket.ts` store 拆分、`ChatList.vue` 等大文件拆分——前提是拆分前后行为完全不变，且有对应测试或人工走查覆盖。
8. 设计令牌落地（硬编码色值替换为 CSS 变量/SCSS 变量）——这是机械的令牌替换，不改变视觉呈现（只要映射关系选对），可以直接执行。
9. `golangci-lint`/`eslint --fix` 类能安全自动修复的存量问题（`gofmt`/`goimports` 对应到前端是 ESLint 的自动修复能力）。

## 2. 必须先出预览/方案、经用户确认后再落地代码（不能拍板全部细节就直接改）

1. **公共页面视觉重构的具体呈现**（`06` 号文档）：断点体系和布局分层原则本身已经在文档里定案，但具体配色方案、是否采纳/如何改造"标准博客风格"闲置组件、卡片视觉细节等，属于产品/审美取舍，必须先用 `frontend-design` skill 出预览（截图或本地 dev server 演示），得到用户确认再写最终代码。这不是"逢视觉必停"——断点数值、响应式分层策略这类已经在 `06` 号文档拍板的技术决策不需要重复确认，只有"看起来是什么样"这类主观判断需要停。
2. **暗色模式下具体视觉细节的取舍**（如某些图标/插图在暗色下的处理方案）：技术路径（用 CSS 变量驱动）已定，具体呈现效果如有明显主观判断空间（例如某个不适配暗色的插图要不要整体替换）需要问。
3. **`websocket.ts` 拆分后 store 的具体命名和字段归属**：`03` 号文档给出的是方向性方案，"未读消息"是否真的应该并入通知 store，需要先读源码确认字段语义,如果和文档假设不一致（比如发现历史上未读消息和系统通知本就是两套独立业务概念，不应该合并），执行者应该按代码实际情况调整方案而不是机械套用文档，这种情况下不需要停下问用户（技术实现细节的合理调整），但如果调整会改变最终 UI 呈现的信息架构（比如把两个原本独立的"未读"入口合并成一个），就回到第 1 条的范畴，需要确认。

## 3. 仍然必须停下问用户（本文档的例外不覆盖这些）

1. `admin-server/scripts/generate-*.sh` 等任何生成脚本的实际执行——即便本轮涉及后端菜单数据修正（`02` 号文档 §2），仍由用户亲自执行。
2. `02` 号文档提到的"菜单 `component` 字段修正 SQL"——这是数据修正 SQL，不是 dev 库脚手架数据，必须停下确认后才能执行，即使是本地/团队 dev 库也不例外（区别于纯新增业务模块的脚手架 SQL，这类是修改现有数据）。
3. 依赖的主版本升级（`vue-tsc`/`typescript`/`vite`/`element-plus` 等）——`07` 号文档已明确本轮不做,如果执行过程中发现某个改造非升级不可,停下来问,不要自行决定升级范围更大的依赖。
4. 任何触及生产配置/部署动作本身——本轮改动范围是 `admin-frontend/` 源码 + 文档，不涉及部署配置，理论上不会触发这条，但如果发现改动会波及 `vite.config.ts` 里的生产 base path（`/admin/`）等部署相关配置，停下确认。
5. AI 自己判断不准的架构/命名/取舍点——拿不准就问，不是逢清单必停，也不是没在清单里就可以自行决定，这条与根规则 `00-workflow.md` 完全一致，本文档不豁免。

## 完成的定义

本文档不是一个需要"完成"的实施任务，而是贯穿 Phase 1-3 全程的判断依据；每次不确定某个具体操作属于第几类时，回来查这张表，拿不准就问。
